### LLM 的限制：智能区 vs 愚蠢区

我想先谈谈 LLM 那些奇怪的限制，这些奇怪的限制是我们许多工作的基础。有一个叫 Dex Hy 的人，他经营着一家名为 Human Layer 的公司，他提出了一个想法：当你使用 LLM 时，它们有一个智能区和一个愚蠢区。当你刚开始使用 LLM，刚刚开始一个新的对话时，你从零开始。那是 LLM 表现最好的时候，因为在那种情况下，注意力关系的压力最小。

每次你向 LLM 添加一个 token，就像给足球联赛增加一支球队。想想每次你给足球联赛增加一支球队时，比赛数量会增加多少。那是二次方增长的。这是因为 token 之间存在注意力关系，这些关系既涉及位置信息也涉及单个 token 的含义。这意味着大约在 40% 左右，或者对我来说大约在 100k 左右——这是我新的标志点——因为不管你是用 100 万上下文窗口还是 200k，情况都一样。它开始变得越来越蠢。所以，当你不断向同一个上下文窗口添加内容时，它会越来越笨，直到做出一些愚蠢的决定。如果这对你来说很熟悉，请举手。是的。酷。

这意味着我们希望将任务控制在智能区范围内，对吧？我们不想让 AI 好高骛远。这其实回到了 Martin Fowler 在重构中提出的老建议，《程序员修炼之道》也提到了这一点。不要贪多嚼不烂。保持任务小规模，这样你作为开发者，作为人类开发者，就不会惊慌失措，不会开始进入愚蠢区。

### 多阶段计划与 Ralph Wiggum 循环

但是如何处理大型任务？如何承担一个大型任务，比如克隆一家公司之类的疯狂事情？如何将其分解为小任务，让它们都能适应智能区？一种方法当然是 AI 公司可能希望你做的，或者自然的做法就是持续不断地进行。你最终会进入愚蠢区，每次请求花费大量 token。然后你 compact 回退。我们稍后会详细讨论 compact。然后你继续，继续，继续，再 compact 回退，继续，继续，继续。我认为这效果不太好，因为沉淀物越多——我们稍后会谈到这一点。

理论是这样的，这也是我一段时间以来在做的：我会使用这些多阶段计划，我会说，好的，我们这里有这个第 4 号任务，这个大型任务。让我们把它分解成小部分，这样我们就可以把它分块处理，每一小块都在智能区完成。如果你曾经使用过多阶段计划，请举手。是的，很常见的做法，对吧？我们一直都是这么做的。当然，在去年十二月之前我也是这么做的。

任何称职的开发者看到这个都会说："这是一个循环，对吧？这是一个循环。我们有阶段一、阶段二、阶段三、阶段四。为什么我们不用阶段 n 呢？阶段 n，我们只需说，好的，我们有一个在后台运作的计划，然后我们在上面循环执行，直到完成。"这就是 Ralph Wiggum 的概念——如果你听说过 Ralph Wiggum 作为一种软件实践，请举手。好的。如果你没听说过 Ralph Wiggum 作为一种软件实践，请举手。实际上，可能更多人是这样的。好的。有一个叫 Ralph Wiggum 的想法，基本上就是基于这个的：你只需要指定旅程的终点——你说好的，我们创建一个 PRD（产品需求文档），来描述我们要去往何方——然后你告诉 AI，做一个小改动，做一个小改动，逐步接近目标。Ralph 的方式还行，但我更喜欢多一点结构。这就是我们在思考智能区方面达到的状态。

我在反驳的、隐含的观点是 specs to code 运动。有人听说过 specs to code 运动吗？请举手。它其实算不上一个运动，只是人们在说 specs to code。它的意思是，有人说，好的，你可以编写一个程序，或者你想构建一个应用。构建这个应用的最佳方式是把一些规格说明写出来，写一份文档，然后把文档变成代码。直接把文档变成代码。怎么做？传给 AI。如果生成的代码有问题，你不去看代码，你回去看 specs，修改 specs，然后继续这样循环。这其实是另一种形式的 vibe coding——你基本上忽略了代码。你不需要担心代码。你只需要不断编辑 specs，最终它会自己搞定。我试过这个。我真的试过，但它很糟糕。它行不通，因为你需要掌控代码。你需要理解代码里有什么。你需要塑造它，因为代码是你的战场。

### 深层模块 vs 浅层模块

在最后这个部分，我要做两件事。首先，我想告诉你们如何——你可能心里有一个问题：假设我有一个我正在工作的代码库，它是一个糟糕的代码库，AI 从来无法在里面做出好的工作——可能实际上大多数人类进入那个代码库也无法做出好的工作——我如何改进那个代码库？其次，我会展示我的并行化设置。

我们先来看糟糕的代码。图表在哪？在这里。在他的书《软件设计哲学》中，John Ousterhout 讨论了模块的理想类型。想象一下你有一个看起来像这样的代码库。每个块都是单个文件。这些文件从它们中导出东西。文件中有你可以拉取然后在其他地方使用的东西。所以你可能有一些奇怪的依赖关系，比如这个文件可能依赖那个文件。

如果这些文件很小，而且没有导出很多东西，John 会把它们称为**浅层模块**——它们看起来像这样。它们基本上就是很多很多小块。这对 AI 来说很难导航，因为它不太理解所有东西之间的依赖关系。它无法弄清楚所有东西在哪里。它必须手动跟踪整个图，说好的，这个依赖这个，这个依赖这个。

而且这些也很难测试。你的测试边界画在哪里？你单独测试每个模块吗？就在这个周围画一个测试边界，然后在下一个周围画另一个测试边界，再下一个？还是你应该做大的分组？你说，好的，我们要一起测试所有这些相关的模块，然后祈求它们能工作。

这意味着，坏的测试看起来大多是这样的——AI 试图把每个小函数都包在自己的测试边界里，然后单独测试它们。但这样做的后果是，假设这个模块调用了那两个模块，它依赖这两者。那么这个模块可能弄错函数的顺序，或者那个模块内部有些东西值得单独测试。如果你在这个模块周围画一个测试边界，你怎么办？你 mock 其他两个模块吗？那怎么搞？所以实际上，弄清楚如何构建一个容易测试的代码库是至关重要的，因为如果我们的代码库容易测试，我们的反馈循环就会更好，AI 就会在我们的代码库中做出更好的工作。说得通吗？

那么好的代码库长什么样？不像那个。它看起来像这样：你有 John Ousterhout 所说的**深层模块**。这些模块有一个小接口，暴露一个小的简单接口，内部有大量功能。这意味着这些很容易测试，因为——假设这个和那个之间有依赖关系。我的箭头能行吗？好了。那么你就在那个模块周围画一个大测试边界，在上面这个周围。你会捕捉到很多好东西，因为你测试了大量功能，而且调用者将有一个简单的接口可以使用。不太复杂。说得通。深层模块 vs 浅层模块。这是好的。这个浅层版本是坏的。

### 改善代码库架构 Skill

你可能会问，我如何把一个看起来像这样的代码库变成一个看起来像那样的？我如何加深模块？嗯，我们有——希望对——一个 skill，叫做"improve codebase architecture"。简洁直接。我们运行它。这个 skill 会扫描我们的代码库，看看有什么可用的。如果你在运行练习，可以自己试试。它在探索架构，探索如何在这个代码库中工作。它会尝试找到加深模块的地方。很简单。

它发现的一个非常酷的东西是，我的课程视频管理器应用的一部分是一个视频编辑器。一个在浏览器中运行的视频编辑器，这很硬核。这是一项不错的工程。我想要一种方法，把整个前端到后端都包装在一个大模块中，这样我就可以测试我在前端按一个按钮，它一直到后端的整个过程。所以我找到了一种方法，通过使用两个类型之间的区分配对联合，我能够使用这个 skill 拥有一个从外部可测试的视频编辑器基础设施的大模块。这意味着 AI 可以看到整个流程，可以对整个流程进行操作，并在整个流程上进行测试。老实说，就 AI 进行更改的能力而言，这是天壤之别，因为如果你不给它好的测试，AI 在视频编辑器上工作是非常残酷的。所以，老实说，如果你今天只带走一件事，那就是在你的仓库上运行这个 skill，看看会发生什么。

### 推送 vs 拉取：编码规范

问题来了——我应该在早期规划阶段尝试优化计划吗？这是我实际上看到很多人在做的事情，这是一个非常好的想法。让我们回到阶段。假设你有所有这些阶段，你和 LLM 一起搞清楚了所有事情。你明白了你要去哪里。你创建了这个旅程/目的地文档。那么你是否应该继续优化再优化那份 PRD，直到它成为你能想象的最完美的 PRD？我认为这没什么价值，因为旅程只是你要去往方向的提示，你需要投入精力的地方是 QA，而且我想你也可以 AFK 做那件事。但根据我的经验，你不会从中获得太多收益——真正重要的是与 AI 的对齐，这是你在最初的 grilling session 中做的。

再问一个问题。还有吗？

（观众："你如何在你的工作流中让它按照你想要的方式编码？这样当你做代码审查时，它至少使用了你想让它使用的库。"）

是的。我们之前其实有过这个问题——你如何对 agent 强制实施你的编码标准？本质上，你如何让它按照你想要的方式编码？有两种不同的方式。**推送**和**拉取**。我所说的推送和拉取是什么意思？推送是你把指令推给 LLM。你说，好的，如果你在 CLAUDE.md 里放点东西——"像海盗一样说话"——这个指令总是会发送给 agent，对吧？所以那是一个推送动作。你在向它推送 token。拉取是你给 agent 一个机会去拉取更多信息。比如 skills。一个 skill 可以放在仓库里，它有一个小描述头，说好的 agent，当你想要时可以拉取这个。

我目前关于代码审查和编码标准的思考是这样的。当你有一个 implementer——怎么回事？好了——implementer。我想让它不那么红。然后你希望编码标准可以通过拉取获得。如果它有疑问，你希望它能回答。但如果你之后有一个自动审查者，那么你希望推送。你想把这些信息推给审查者。你想说："这些是我们的编码标准。确保这段代码遵循它们。"所以如果你有 skills，那么你想把这些东西推给审查者，这样审查者既有已编写的代码，又有编码标准可以比较。希望这回答了你的问题。我也可以给你看一个自动化的版本。

### Sand Castle：并行 AFK 循环

实际上，是的，我们趁现在还记得的时候来做。我最近花了一周左右的时间构建了一个叫做 **Sand Castle** 的东西。对于市场上的 AFK agent 运行选项我不太满意。它本质上是一个用于运行这些循环的 TypeScript 库。你有一个 run 函数，创建一个 worktree，在 Docker 容器中沙箱化它，然后允许你在里面运行一个 prompt。在那个 worktree 中，它就是一个 git 分支，你有那个代码，之后可以合并它。

如果我打开——有一些非常漂亮的查看方式——它本质上允许你运行这种自动循环，并且允许你非常简单地跨多个不同 agent 进行并行化。我进入我的 sand castle 文件，到 `main.ts`，我们来看一下。这有点像之前我给你们看的 Ralph 循环的一个版本。这是我们从顺序到并行的转变。

我们首先有一个 planner，它接受一个计划 prompt，查看 backlog，选择一定数量的要并行处理的 issue。还记得我之前给你看的那个有阻塞关系的看板吗？它计算出所有阶段。所以它会说，好的，假设我们有——你可以忽略这里所有的胶水代码。这基本上就是一组 issue，GitHub issue，带有一个标题和一个分支供你工作。然后对于每个 issue，我们创建一个 sandbox，然后在该 sandbox 中运行 implement，传入 issue 编号、issue 标题和分支。这就像我们之前运行的循环。

如果它创建了一些提交，我们然后审查这些提交。这基本上就是循环。我们如何处理这些提交？我们将它们传入一个 merger agent，它接受一个合并 prompt，接受已创建的分支，接受 issue，然后合并它们。如果合并有任何问题——类型、测试之类的问题——它来解决。

这已经是我在大多数项目上的工作流有一段时间了。它运作得非常好。是的，我建议你去看看 Sand Castle 了解更多。为了正确回答你的问题，在审查者中，我会推送编码标准，在 implement 中，我允许它拉取。我实际上在实现中使用 Sonnet，在审查中使用 Opus，因为我认为审查——我需要更聪明的模型。
