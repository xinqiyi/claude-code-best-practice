[`3:03`](https://youtu.be/PQU9o_5rHC4?t=183) 这个想法一直存在，实现它的方式是：你先教模型如何编程，然后教它如何使用工具，再教它如何使用计算机。你可以看到这一点，因为我加入 Anthropic 时加入的第一个团队叫做 Anthropic Labs 团队，它产出了三个产品：Quad Code、MCP 和 desktop app。所以你可以看到这些是如何交织在一起的。具体到我们构建的这个产品，没有人要求我构建一个 CLI。我们隐约知道也许是时候构建某种编程产品了，因为模型似乎已经准备好了，但还没有人真正构建出能够利用这种能力的产品。那种产品过剩的感觉仍然很强烈。但在当时，这种感觉更加强烈，因为还没有人做出这种东西。于是我开始了各种尝试，我想，"好吧，我们要构建一个编程产品。我首先要做什么？我必须先学会如何使用 API，因为我那时还没用过 Anthropic 的 API。" 所以我只是构建了一个小型的终端应用来使用 API。仅此而已。那是一个小小的聊天应用，因为你知道当时的 AI 应用，对于非程序员来说，大多数人使用的就是一个聊天应用。所以我就是那么做的。它在终端里。我可以提问，它给出答案。后来 tool use 功能发布了，我只是想试试它，因为我不太理解这是什么。我想，"tool use 很酷，但这真的有用吗？可能没有用吧。让我试试看。"

[`6:09`](https://youtu.be/PQU9o_5rHC4?t=369) 是的，这是个意外。我记得在终端开始在内部流行起来之后。老实说，在构建这个东西之后，第一个原型大概两天后我就开始把它给我的团队做内部测试，因为如果你想到一个想法，而且它看起来有用，你第一件事就是把它交给别人，看看他们怎么用。第二天我来上班，坐在我对面的 Robert，他是另一位工程师，他把 Quad Code 装在了自己的电脑上，正在用它编程。我说，"你在干什么？这东西还没准备好。它只是一个原型。" 但没错，在那个形态下它已经有用了。我还记得我们做发布评审、准备对外发布 Quad Code 的时候，那是 2024 年 12 月还是 11 月左右。Dario 问了一个问题，他说，"内部的使用图表简直是垂直上升的。你是不是在强迫工程师们用它？你为什么强制要求他们使用？"

[`7:00`](https://youtu.be/PQU9o_5rHC4?t=420) 我说，"不，不，我们没有。我只是发了条消息，他们就开始互相推荐了。" 老实说，这完全是偶然的。我们从 CLI 开始，因为它是最便宜的方式，然后它就一直停留在那里了。

[`7:13`](https://youtu.be/PQU9o_5rHC4?t=433) 那么在 2024 年那段时间，工程师们是怎么使用它的？他们已经开始用它来交付代码了吗，还是以不同的方式使用？

[`7:19`](https://youtu.be/PQU9o_5rHC4?t=439) 模型当时在编程方面还不太好。我个人用它来自动化 git 操作。我觉得我大概率已经忘了大部分 git 命令，因为 Claude Code 已经帮我做了太久了。但是，自动化 bash 命令是一个很早的用例，还有操作 Kubernetes 之类的东西。也有人用它来编程，所以有一些早期迹象。我认为第一个用例其实是写单元测试，因为风险比较低，而且模型当时还相当差。

[`7:46`](https://youtu.be/PQU9o_5rHC4?t=466) 但人们正在摸索如何使用这个东西。

[`7:51`](https://youtu.be/PQU9o_5rHC4?t=471) 我们还看到了一件事——人们开始为自己编写 markdown 文件，然后让模型读取这些 markdown 文件。这就是 CLAUDE.md 的由来。对我来说，产品设计中最重要的原则可能就是潜在需求。这个产品的每一点，都是在最初的 CLI 之后，通过潜在需求构建出来的。CLAUDE.md 就是一个例子。还有一个我觉得可能很有趣的通用原则：你可以为模型构建东西，然后在模型周围搭建脚手架来稍微提升性能，根据领域不同，你可能能提升 10% 到 20%。然后这些提升基本上会被下一个模型抹平。所以你要么搭建脚手架，获得一些性能提升，然后重新搭建；要么你直接等下一个模型，就能免费获得这些提升。CLAUDE.md 和这类脚手架就是这样的例子。我认为我们之所以一直留在 CLI 里，是因为我们觉得没有任何 UI 是我们在六个月后仍然相关的，因为模型进步得太快了。

[`8:50`](https://youtu.be/PQU9o_5rHC4?t=530) 刚才我们说到应该比较一下 CLAUDE.md，但你说了一句很深刻的话——你的 CLAUDE.md 其实非常短，这和人们预期的可能完全相反。为什么呢？你的 CLAUDE.md 里写了什么？

[`9:03`](https://youtu.be/PQU9o_5rHC4?t=543) 好的，我来之前检查过。我的 CLAUDE.md 只有两条。嗯，只有两行。第一行是：每当你提交一个 PR，启用自动合并。这样一旦有人批准，它就会自动合并。这样我就可以继续写代码，不需要在代码评审上反复折腾。第二行是：每当我提交一个 PR，把它发到我们内部的团队 Slack 频道。这样有人可以快速审批，我就能继续推进。其他的所有指令都在我们代码库里的 CLAUDE.md 中，那是我们整个团队每周多次共同贡献的内容。我经常看到有人提交 PR，犯了一些完全可以避免的错误，我就会直接在 PR 上 @Claude。我会说，"把这个加到 CLAUDE.md 里"，每周都会这样做很多次。

[`9:52`](https://youtu.be/PQU9o_5rHC4?t=592) 你需不需要压缩你的 CLAUDE.md？我确实到了那个阶段——在顶部看到一条消息说我的 CLAUDE.md 已经有几千个 token 了。你们遇到这个问题时会怎么做？

[`10:03`](https://youtu.be/PQU9o_5rHC4?t=603) 我们的 CLAUDE.md 其实相当短。我想大概也就几千个 token。如果你遇到这个问题，我的建议是删掉你的 CLAUDE.md，重新开始。

[`10:11`](https://youtu.be/PQU9o_5rHC4?t=611) 有意思。

[`10:12`](https://youtu.be/PQU9o_5rHC4?t=612) 我觉得很多人试图过度设计这个东西。但其实能力随着每个模型都在变化。你想要做的是用最小可行的方式让模型走在正确的轨道上。所以如果你删掉 CLAUDE.md，然后模型走偏了，做了错误的事情，那时候你再一点一点加回去。你可能会发现，随着每个模型的进步，你需要加的东西越来越少。对我来说，老实说，我认为自己是一个相当普通的工程师。我不使用很多花哨的工具。我不用 Vim。我用 VS Code，因为它更简单。我真的不——

[`10:44`](https://youtu.be/PQU9o_5rHC4?t=644) 等等，真的吗？我以为因为你在终端里构建了这个东西，你会是一个死忠的终端用户，只用 Vim 的那种人，"去他的 VS Code 用户"那种。

[`10:53`](https://youtu.be/PQU9o_5rHC4?t=653) 嗯，团队里有这样的人——比如 Adam Wolf，他在团队里，他就说，"你永远不可能从我冰冷的手里夺走 Vim。" 所以团队里确实有很多这样的人。这是我早期学到的一件事——每个工程师喜欢以不同的方式使用他们的开发工具，他们喜欢使用不同的工具，没有一种工具适合所有人。但我也认为，这也是 Claude Code 能够如此出色的原因之一，因为我思考的是：什么是我会使用、对我有意义的产品。所以使用 Claude Code，你不必懂 Vim，不必懂 tmux，不必知道如何 SSH，不必知道所有这些东西。你只需要打开这个工具，它会引导你，它会帮你做所有这些事情。

[`11:30`](https://youtu.be/PQU9o_5rHC4?t=690) 你是如何决定终端输出的详细程度的？有时候你需要按 Ctrl+O 来查看。内部有没有关于应该更详细还是更简短的争论？每个用户可能都有自己的看法。你是如何做这些决定的？

[`11:47`](https://youtu.be/PQU9o_5rHC4?t=707) 你的看法是什么？现在的输出是不是太详细了？

[`11:50`](https://youtu.be/PQU9o_5rHC4?t=710) 哦，我喜欢这种详细程度，因为有时候它就像失控一样，我在看着，可以很快地阅读，然后说，"哦不，不对。" 然后我按 Escape 键停止它，它就在 bug 还在生产的过程中把它扼杀了。我的意思是，那通常是我没有正确使用 plan mode 的时候。

[`12:07`](https://youtu.be/PQU9o_5rHC4?t=727) 这是我们经常改变的东西。我记得早期，大概是六个月前，我试图去掉 bash 输出，只是总结一下，因为我觉得那些又长又大的 bash 命令，我并不真的关心。然后我给 Anthropic 的员工用了一天，所有人都反对。"我要看我的 bash 输出"，因为它实际上非常有用。对于 git 输出可能没什么用，但如果你在运行 Kubernetes 任务之类的，你确实想看到它。我们最近隐藏了文件读取和文件搜索的输出。你会发现它不再显示"read foo.md said"，而是显示"read one file, searched one pattern"。我认为这在六个月前是做不到的，因为模型还没有准备好。它仍然经常会读错东西。作为用户，你仍然需要在旁边盯着，发现问题并调试。但现在我发现它几乎每次都在正确的轨道上。因为它使用工具如此频繁，实际上总结一下会更好。但我们发布之后，内部 dogfood 了一个月，GitHub 上的用户不喜欢它。有一个很大的 issue，人们说"不，我想看细节"，那是非常好的反馈。所以我们添加了一个新的 verbose mode，在 `/config` 里你可以启用 verbose mode，如果你想看到所有的文件输出，你可以继续看。我在那个 issue 上回复了，但人们仍然不喜欢，这同样很棒，因为我最喜欢的事情就是听到用户的反馈，了解他们真正想怎么使用它。所以我们不断地迭代，把它做得更好，做出人们想要的东西。我很惊讶我现在有多么享受修 bug。你只需要有非常好的日志记录，然后甚至只需要说"嘿，看看这个特定的对象，它在这个地方出错了"，它就会搜索日志，找出所有问题。它可以创建一个生产隧道，帮你查看生产数据库。这太疯狂了。修 bug 就是：去 Sentry，复制 markdown。很快它就会直接用 MCP 了。就像一个自动修 bug 和自动写测试的——他们管这个叫什么来着？"制造创业公司的工厂"？哦对。

[`14:10`](https://youtu.be/PQU9o_5rHC4?t=850) 没错。现在有很多这样的概念——不再需要评审代码。我是个老派的人，所以我喜欢详细输出。我喜欢说，"哦，你在做这个，但我想要你做那个。" 但现在有了一种完全不同的思路，说任何时候人类需要看代码都是不好的。

[`14:30`](https://youtu.be/PQU9o_5rHC4?t=870) 是的。是的。是的。
