[`0:02`](https://youtu.be/AmdLVWMdjOk?t=2) 模型进化得太快了。如果你在 3 个月或 6 个月后问我这个问题，我的答案会完全不同。这位是 Boris Cherny，他是 Claude Code 的创造者，前 Meta 首席工程师。我们聊了塑造他职业生涯的一切。你能解释一下什么是"潜在需求"吗？

[`0:17`](https://youtu.be/AmdLVWMdjOk?t=17) 我认为潜在需求是产品领域中唯一最重要的原则。

[`0:22`](https://youtu.be/AmdLVWMdjOk?t=22) 你说过存在一些明显的文化差异，这很困难。

[`0:26`](https://youtu.be/AmdLVWMdjOk?t=26) 天哪，说"困难"简直是太轻描淡写了。那简直就是一场噩梦。我们还聊到了 Claude Code，以及 Anthropic 内部正在发生的事情。尽管 Anthropic 的人员规模增加了两倍，但每位工程师的生产力却增长了近 70%，这都得益于 Claude Code。不要为今天的模型构建产品，要为 6 个月后的模型构建。我会向所有人推荐一本对我作为工程师影响最大的技术书籍。你对与 Codex 和 OpenAI 的竞争有什么看法？我想从你职业生涯的开端聊起——你在 Meta 晋升为高级工程师的经历。是什么项目让你获得了晋升，当时你在做什么？如果我没记错的话，那个项目是"聊天与群组"，目的是让 Messenger 和 Facebook 更紧密地结合在一起。实际上，我在 Meta 最初做的几个项目都与 Messenger 和 Facebook 有关。第一个项目大概是 Zuckerberg 的一个想法，关于同步 Messenger 聊天和 Facebook 群组。我们做了好几个这样的项目，试图让 Messenger 和 Facebook 更加融合。动机是大家感觉到这种公共空间社交产品正在消失，事情正在向聊天和更随意、更实时的空间转移。所以我们尝试了几个版本的产品，"聊天与群组"是成功的那一个。我记得大概是第三或第四个版本。我当时在 Facebook 群组部门（或者说在 Facebook），与 Messenger 的合作在组织架构上距离很远。这个想法是当时的 PM Steve 提出来的，他觉得这是我们该做的东西。我抓住了这个想法，然后说，太好了，我们干吧。于是我就开始动手做。很快就有了一些起色，所以我申请了更多工程师，有三位工程师加入——Shatambri、Crystal 和 Chaang，他们是第一批加入这个项目的工程师。之后我们又得到了数据科学和设计方面的支持。项目从 Web 开始，后来也拓展到了移动端。我们证明了这个想法是可行的——你可以在 Facebook 群组里嵌入聊天功能，这种产品模式是行得通的。说实话，当时有很多地方做得并不好。以现在的产品标准来看，体验非常粗糙。那个年代大家都在 Web 上构建产品，出现各种 bug 是完全被接受的。如今，无论是视觉标准还是质量标准都高了很多。产品在不断增长，而我们团队很小，每个人都得做所有事情。我记得我们没有用户研究员，所以午餐时间我会去食堂，拿着新功能给食堂工作人员看，问他们能不能找到打开聊天的方式。有时他们能找到，有时找不到。这就是一种观察式的用户研究——你看人们在特定情境下如何完成一项任务，不要给太多提示，观察他们在哪里遇到困难，他们理解了哪些东西。后来我把这个方法教给了团队，很快我们所有人都会在午餐时间去食堂"打扰"工作人员，把他们当作代表性用户，问他们这个功能是否合理。有趣的是，早期 Facebook 的文化让工程师能做代码之外这么多事情。比如你在做用户研究，我记得你还做过设计，还指导别人做设计。这真是 Facebook 文化中一个非常独特的地方。我认为这一点非常重要，直到今天，在 Claude Code 团队——也就是我现在所在的团队——我们仍然非常重视通才。我喜欢和通才一起工作。如果你是一个会写代码、同时也能做产品工作、能做设计、有产品直觉、愿意去和用户交流的工程师，我就喜欢和这样的人合作。现在我们所有职能岗位都是按这个标准招聘的。我们的产品经理会写代码，数据科学家会写代码，用户研究员也会写一点代码。我就喜欢这样的通才。这其实是我成长的方式——从 18 岁开始运营第一家初创公司时，我就什么都得做。在加入 Facebook 之前，我在小公司工作过，什么都得做。我觉得在大公司里，你往往会被迫进入一个特定的"泳道"，但这其实是人为的，因为工程到底是什么？它是一个非常狭窄的技能集合，但你实际上在做的是构建产品或基础设施，而要把一件事从头到尾做好，远不止写代码那么简单。能在 Facebook 这样的地方工作真的很酷，因为 Facebook 在当时确实会奖励这种行为。实际上，在那个半年度考核结束时我得到了晋升，而下一个半年度，当时加入的所有工程师也都晋升了。在那些早期产品中，你多次提到一个概念叫"潜在需求"，这似乎是很多产品方向的推动力。你能解释一下什么是潜在需求吗？

[`5:42`](https://youtu.be/AmdLVWMdjOk?t=342) 我认为潜在需求是产品领域中唯一最重要的原则。你看看 Facebook 那些成功的产品，每一个都包含潜在需求的元素。比如 Marketplace，它源于这样一个观察：如果你看当时的 Facebook 群组，40% 的帖子都是在买卖东西。Facebook 群组并非为商业交易而设计，但人们就是用它来做这个。这很酷——你设计一个产品时让它有一定的可被"黑客"利用的空间，用户会稍微"滥用"它，然后你观察数据，看看他们是如何滥用的，再围绕这个来构建产品。先是 Facebook 群组，然后出现了买卖群组，这显然成功了，因为人们本来就希望在 Facebook 群组里买卖东西，Marketplace 是下一步的自然延伸。它只是人们已有意图的自然拓展。Facebook  Dating 也很类似。观察结果是大约 60% 的个人主页浏览来自非好友关系的异性。这就像是传统的"互相窥视"，对于当时的 Nate 和 FMS 来说，这就是这个产品会成功的证据。产品领域的原则是：你永远无法让人们去做他们还没在做的事情。你能做的是找到他们已有的意图，然后引导他们更好地利用这个意图，让他们更轻松地做他们想做的事。在你的故事里，你还提到你在跨组织协作。你在搭建 Messenger 和群组工程之间的桥梁。我很好奇，你说存在一些明显的文化差异且很困难，对于跨不同文化组织的工作你有什么建议吗？

[`7:30`](https://youtu.be/AmdLVWMdjOk?t=450) 天哪，说"困难"简直是太轻描淡写了。那简直就是一场噩梦。当时的 Facebook 想要快速发布产品，尽快推出优秀的产品。而 Messenger 则只关心可靠性和性能。这是完全相反的价值观。这不只是文化层面的，也不仅仅是工程师之间的差异。Messenger 团队的工程师们对我们持怀疑态度，因为我们会影响他们的性能指标。在组织架构上，他们的目标是慢慢发布产品，不使指标倒退；而我们的目标是快速发布。目标也完全不同——他们有 SLA 可用性指标，而我们只关心日活跃用户和参与度。所以我的体会是，这些文化价值观非常根深蒂固。这不仅仅是人们挂在嘴上的东西，你实际上可以在组织设计、目标设计和方方面面都看到它。老实说，那个项目失败的原因之一——虽然它后来演变成了成功的东西，但那个版本是失败的——就是因为价值观的差异。所以我认为，如果你想让价值观截然不同的团队成功合作，你必须找到某种共同的目标、共同的利益、共同的信念，或者某种双方都想共同验证的假设——如果成功了对双方都很有意思。而"聊天与群组"这个项目从根本上说对 Facebook 很酷，但出于很多原因对 Messenger 来说并没有那么酷。以你现在的经验，如果回到那个项目，你会怎么做？

[`9:10`](https://youtu.be/AmdLVWMdjOk?t=550) 我可能会直接去找 Zuckerberg，跟他说，如果你真的认真对待这件事，我们应该把 Messenger 并入 Facebook 的组织架构。后来这确实发生了，而且还发生了好几次——Messenger 在组织里，然后移出去，又移进来，又移出去。大公司就是这样。但我觉得要让这种事情成功，共同的汇报人不能是 Chris Cox 这种级别，应该再低一点，这样才能让组织架构更具协作性。

[`9:39`](https://youtu.be/AmdLVWMdjOk?t=579) 明白了，就是为了对齐激励机制，避免那种持续的拉扯。

[`9:44`](https://youtu.be/AmdLVWMdjOk?t=584) 对，完全正确。在你的职业生涯中，我看到你做过很多有趣的副项目，我很好奇这些项目带来了什么样的"蝴蝶效应"？比如在你加入 Meta 之前，你做过 Undux，一个 React 的状态管理框架。这对你的职业生涯有什么影响吗？

[`10:07`](https://youtu.be/AmdLVWMdjOk?t=607) 对我来说，副线项目非常重要。我招聘工程师时，这绝对是我会关注的一点。我希望看到有副线项目的人——酷炫的周末项目、有趣的副项目，甚至是一个特别热衷于做康普茶之类东西的人。你希望和那些对主业之外的事物充满好奇心的人一起工作。他们都是全面发展的人，是我喜欢共事的人。对我来说，我的很多成长正是来自这些副项目。比如 Undux，它的来历是因为 React 的状态管理真的过于复杂了。当时的"最先进"方案是 Flux，后来又有 Redux，但我就是搞不懂 Redux。我认为自己是个普通工程师——我构建产品，我不是那种了不起的系统工程师。当时的 Redux 有 reducer 这些概念，更新一点状态就要经过非常复杂的流程，我就是理解不了。所以我构建了一个更简单的替代品，看起来能用。那时我在一个非营利组织做志愿者，他们开始用这个，他们的工程师很喜欢。后来我加入 Facebook，看到很多人在使用 Redux 时遇到同样的问题——公司内部有一个 Redux 用户群，里面全是我曾经问过的那种问题。当你作为工程师或产品人员遇到一个问题时，有时候只是你一个人遇到，但往往其他人也遇到了。培养这种"蜘蛛感应"很重要——判断这个问题是否也可能困扰着别人。这绝对是一个普遍存在的问题，我从技术支持帖子和团队使用 Redux 的困难中就能看出来。于是我在内部发布了 Undux。Undux 还行，不是特别出色的产品，但至少比 Redux 好。在 Facebook，我其实不知道如何推广它。我先发了帖子，一些人开始使用。我记得通知团队的 Jeff Case 是早期重度用户，我们花了很多个深夜调试一些棘手的通知相关 bug。我想要更多用户，于是写了一个小脚本，抓取了报告问题的用户群，按团队统计，然后通过聊天联系每个团队的技术负责人和经理，为每个团队安排了一场技术讲座。前后大概做了 20、30 到 40 场讲座，持续了几周时间。

[`12:45`](https://youtu.be/AmdLVWMdjOk?t=765) 我记得骑着自行车在 Meta 园区里穿梭做这些讲座，非常有趣，因为大家都很投入，他们很高兴有人真的关心并解决他们面临的问题。后来 Undux 一度成为 Facebook 最流行的状态管理框架，但很快被 Recoil 和更现代的替代方案取代了，现在是 Relay 之类的技术。这种副项目会出现在你的绩效评估中吗？或者对你的晋升有帮助吗？它确实出现在了我的绩效评估里。以 Meta 的标准来说，它更像是锦上添花，并不是能让你直接晋升到下一级别的关键因素。但那段时间我还有很多其他副线项目。比如有段时间我迷上了 TypeScript，那是在我上一家公司。我们当时在用 TypeScript，但好的学习资源很少。我就开始写一本关于它的书，因为我觉得总得有人来做这件事——太疯狂了，竟然没有这样的书。这门语言太棒了，设计极其出色，包含了很多当时其他语言没有的概念。当时还没有条件类型，但像条件类型、字面量类型、映射类型等等，这些都是非常厉害的特性，连最资深的 Haskeller 都会印象深刻。但没人写这方面的内容，所以我深入钻研并写了这本书，它占据了我差不多一年的时间。（笑）我不推荐这么做，但深入研究真的很有趣。后来我还在旧金山创办了当时全球最大的 TypeScript 聚会。那是个很好的机会，我遇到了 Node.js 的创造者 Ryan Dahl，还有很多著名的 JavaScript 名人。这让我意识到，这些人也都是普通人，每个人都在做很酷的东西，有些在当时很酷，但归根结底大家都是普通人，任何人都能做到这些。

[`14:47`](https://youtu.be/AmdLVWMdjOk?t=887) 你后来在 Meta 或者甚至在 Anthropic 有用到 TypeScript 或那些技术深度吗？有的，说来有趣。我以前其实不太在意编程语言。大概 10 年前，我骑摩托车出了很严重的车祸，双臂骨折了。

[`15:05`](https://youtu.be/AmdLVWMdjOk?t=905) 天哪。

[`15:08`](https://youtu.be/AmdLVWMdjOk?t=908) 天哪，那你怎么写代码？

[`15:10`](https://youtu.be/AmdLVWMdjOk?t=910) 这就是问题所在。我大概一个月不能写代码，手还在疼，没法写我当时用的 JavaScript。所以我不得不去学其他语言，因为它们用的击键次数更少。我先从 CoffeeScript 开始，因为它括号更少。我觉得这门语言现在可能已经不存在了，没人用了。但这也是我接触 Haskell 和函数式编程的契机，因为用更少的击键可以做同样的事情，这就是当时的动机。后来我在一家对冲基金工作——那是在 Facebook 之前——有个同事 Rick 非常喜欢 Scala，我当时不太理解 Scala，他把我带进了函数式编程的世界。这就是那本我会推荐给所有人的技术书籍——《Scala 函数式编程》。它对我作为工程师的影响最大。你可能永远不会用 Scala，但它教你思考编程问题的方式，与大多数人以往在学校或实践中的思维方式完全不同，非常不可思议，它会彻底改变你写代码的方式。对我来说，先是 Haskell 和 CoffeeScript 这几个关键语言作为第一步，然后是 Scala，再然后是 TypeScript。这改变了我的思维方式，现在我在写代码时会用类型来思考——你的代码中最重要的东西是类型签名，这比代码本身更重要。把类型搞定，代码就会非常清晰。所以即使在 Facebook 我主要写 Flow 和 Hack，后来在 Instagram 写 Python，这些都很有帮助。在 Anthropic 我主要写 TypeScript 和 Python，所以实际上非常相关。但更大的教训是：用类型来思考。

[`17:07`](https://youtu.be/AmdLVWMdjOk?t=1027) 在你职业生涯的这个阶段，你提到自己入职时被定级偏低了，你以中级工程师的身份加入，尽管你有很多经验。你说事后看来，被定级偏低其实是幸运的。你是怎么想的？

[`53:21`](https://youtu.be/AmdLVWMdjOk?t=3201) Unshipping 的意思是：如果你只是给 App 增加功能，对一小部分用户来说很酷，但对大部分不用该功能的用户来说反而是坏事。想象一个只增加功能的 App，随着时间的推移，功能不断累积。如果每个功能只有 10% 的人使用，普通用户看到的就是一大堆他们不用的功能。App 看起来杂乱无章、令人困惑，用户打开时不知道该做什么。软件的本质是屏幕大小有限，这是一个有限的资源，所有功能都在竞争这个资源。增加一个功能就意味着占用了用户本可以使用的另一个功能的机会。所以 unshipping 的意思是：你必须设定一个使用量标准，如果一个功能达不到这个标准，就把它删除。一小部分用户会不高兴，但对大多数用户来说非常好，从整体来看对所有人都有好处。
