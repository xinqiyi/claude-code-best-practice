### 开场介绍

我们非常激动迎来我们的首位特邀嘉宾。他帮助构建了现代 AI，然后解释了现代 AI，再然后偶尔重新命名了现代 AI。他实际上就在这间办公室共同创立了 OpenAI。当年也是他在 Tesla 让 Autopilot 真正跑了起来。他有一种罕见的天赋，能让最复杂的技术变革既显得可理解又显得不可避免。大家认识他是因为他去年创造了"vibe coding"这个词，但就在过去几个月里，他说了一些更加令人震惊的话——他说他从未感到自己作为程序员如此落后。这就是我们今天要聊的起点。感谢 Andrej 加入我们。

是的。大家好。很高兴来到这里，也很高兴为我们开场。

### "从未感到作为程序员如此落后"

好的。就在几个月前，你说你从未感到自己作为程序员如此落后。这话从你嘴里说出来真是令人震惊。你能帮我们解读一下吗？这种感受是令人振奋还是令人不安？

当然是两者皆有。首先，和许多人一样，我一直在使用像 Claude Code 这样的 agentic 工具以及类似的东西，可能已经用了一年左右。这些工具在处理代码块时表现很好，但有时也会出错，你需要去修改它们，算是有帮助吧。然后我想说，去年十二月是一个明显的转折点——那时我在休假，所以有更多时间。我想很多人也是如此——我开始注意到，使用最新的模型，代码块直接就能用了，然后我继续要求更多功能，它们也都能直接用了，然后我不记得上次我纠正它是什么时候了。然后我越来越信任这个系统，于是我就开始 vibe coding 了。

这是一个非常显著的转变。我想很多人实际上——我在推特（或者说 X）上试图强调这一点——因为我觉得很多人去年对 AI 的体验还停留在 ChatGPT 那个层面。但你真的需要重新审视一下，你需要从十二月开始重新审视，因为情况已经发生了根本性的变化——尤其是这种 agentic 的连贯工作流真正开始奏效了。所以我可以说，正是这种认识让我彻底陷入了无限 side projects 的兔子洞。我的 side projects 文件夹里塞满了各种随机的项目，我一直在不停地 vibe coding。所以这大概发生在去年十二月，从那以后我一直在关注这件事的影响。

### Software 3.0：新的计算范式

你多次谈到 LLM 作为一种新计算机的观点。这不只是更好的软件，而是一整套新的计算范式。Software 1.0 是显式规则，software 2.0 是学习得到的权重，software 3.0 就是现在这个。如果这真的是事实，那么一个团队从真正相信这一点的那一天起，会在哪些方面做出不同的建设？

没错。Software 1.0 是我在写代码，software 2.0 是我通过创建数据集和训练神经网络来编程。所以编程有点像在整理数据集，加上一些目标函数和神经网络架构。然后发生的事情是，基本上如果你在足够多的任务集上训练这些 GPT 模型或 LLM——隐式地，因为通过训练互联网数据你必须多任务处理数据集中的所有内容——这些模型在某种意义上就变成了一个可编程的计算机。

所以 software 3.0 大概就是，你的编程现在变成了提示（prompting），而上下文窗口中的内容就是你操控解释器（即 LLM）的杠杆，这个 LLM 解释你的上下文并在数字信息空间中进行计算。

我认为有几个例子真正让我理解了这一点。例如，当 OpenClaw 发布时，如果你想安装 OpenClaw，通常你会认为应该有一个 bash 脚本，一个 shell 脚本。所以运行 shell 脚本来安装 OpenClaw。但问题是，为了支持多种平台和多种类型的计算机，这些 shell 脚本通常会变得极其复杂。但你仍然被困在 software 1.0 的思维里，想要写代码。实际上，OpenClaw 的安装就是一段你可以复制粘贴给你的 agent 的文本。所以基本上就是一个小的 skill："复制粘贴这个给你的 agent，它会帮你安装 OpenClaw。" 这之所以强大得多，是因为你现在工作在 software 3.0 的范式下，你不需要精确地列出所有的安装细节。agent 有自己的智能，它会打包这些指令，查看你的环境、你的计算机，然后执行智能操作来让一切正常运行，并在过程中进行调试。这强大得多了。所以这是一种完全不同的思考方式——什么是可以复制粘贴给你的 agent 的那段文本？这就是编程范式。

### MenuGen 和"本不该存在的应用"

另一个更极端的例子是我在构建 MenuGen 的时候。MenuGen 的想法是这样的：你去餐厅，服务员给你菜单。通常菜单上没有图片。所以我不知道这些东西是什么——通常 30% 的东西我不知道是什么，甚至 50%。所以我想给餐厅菜单拍张照片，然后得到这些菜品在一般意义上可能长什么样的图片。

我用 vibe coding 做了一个应用，基本上可以让你上传一张照片，它做各种处理，运行在 Vercel 上，它重新渲染菜单，给你展示所有菜品，并为每个菜品生成一张图片——它用图像生成器来 OCR 所有不同的标题，用图像生成器获取它们的图片，然后展示给你看。

然后我看到了这个的 software 3.0 版本——这让我震惊——就是直接拿你的照片，给 Gemini，然后说"用 Nanobanana 把东西覆盖到菜单上"。Nanobanana 基本上返回了一张和我拍的菜单完全一样的图片，但它实际上在像素中渲染了菜单上的不同内容。这让我震惊，因为实际上我所有的 MenuGen 都是多余的。它是在旧范式下工作的。这个应用本不该存在。

Software 3.0 的范式更加原始。你的神经网络承担了越来越多的工作，你的 prompt 或上下文就是那张图片，输出也是一张图片，根本不需要中间的应用。所以人们需要重新思考——不要停留在已有的事物范式里，不要仅仅把它看作是对现有事物的加速。现在有了新的可能性。

回到你关于编程的问题——我认为这也是一个在旧思维模式下工作的例子，因为这不仅仅是编程和编程变快了。这是更通用的信息处理，现在可以自动化了。所以这不仅仅是代码的问题。以前的代码处理的是结构化数据，对吧？你在结构化数据上写代码。但比如我的 LLM 知识库项目——基本上是用 LLM 为你的组织或为你个人创建 wiki——这甚至不是一个程序。这在以前是不可能存在的，因为没有代码能基于一堆事实创建知识库。但现在你可以直接拿这些文档，用不同的方式重新编译它们，重新组织它们，创造出一些新的、有趣的东西，作为数据的重新构架。这些都是以前不可能的新事物。所以这是我一直在强调的一点——不仅仅是现有的东西能做得更快，而且还有以前根本不可能做到的新机会，我认为这更令人兴奋。

### 展望 2026 年及以后

我很喜欢你刚才描述的 MenuGen 的演进和二分法。如果再进一步推演，2026 年的"在 90 年代建网站"、"在 2010 年代做移动应用"、"在上一波云时代做 SaaS"的对应物是什么？什么东西在事后看来会很显然，而今天大部分还没有被构建？

以菜单为例，很多代码本不该存在，应该由神经网络完成大部分工作。我认为这种推演看起来会非常奇怪，因为你基本上可以想象在某种意义上完全神经形态的计算机。你输入原始视频——想象一个设备，它接收原始视频或音频，直接进入一个神经网络——然后使用扩散模型渲染一个在那一刻独一无二的 UI。

我感觉在计算早期，人们其实有点困惑，不知道计算机会像计算器还是像神经网络。在五六十年代，哪种方向会胜出并不明显。当然我们走了计算器的路，最终建立了经典计算，而神经网络目前是在现有计算机上虚拟化运行的。但我认为这一切将会翻转，神经网络成为宿主进程，CPU 成为协处理器。所以我们看到神经网络智能计算的图表将占据主导地位，成为 FLOPs 的主要消耗。所以你可以想象一些非常奇怪和陌生的场景，神经网络承担大部分繁重工作。它们将工具使用（tool use）作为历史遗留的附属功能来处理某些确定性任务。但真正主导的是这些神经网络。所以你可以想象极其陌生的推演结果，但我认为我们可能会一步一步地到达那里。

### 可验证性与锯齿状智能

我想聊一下可验证性这个概念——AI 会更快速、更容易地自动化那些输出可以被验证的领域。如果这个框架是对的，那么哪些工作会以远超人们预期的速度发生变化？又有哪些人们实际认为安全、但实际上高度可验证的职业？

我花了一些时间写关于可验证性的文章。基本上，传统计算机可以轻松自动化你能用代码精确描述的事情，而这一轮 LLM 在某种意义上可以轻松自动化你能验证的事情，因为其工作方式是：前沿实验室在训练这些 LLM 时，用的是巨大的强化学习环境。它们被给予验证奖励，然后由于这些模型的训练方式，它们最终不断进步，创造出这些锯齿状的实体——在数学、代码等可验证领域能力达到顶峰——而在不在此空间内的领域则表现平平、粗糙。

所以我写可验证性，是因为我想理解为什么这些东西如此锯齿状。部分原因与实验室训练模型的方式有关，但我认为部分原因也与实验室的侧重点以及他们碰巧放入数据分布的内容有关。因为有些事情在经济上明显更有价值，最终创建了更多环境，因为实验室希望在这些场景下工作。所以我认为代码是一个很好的例子。可能还有很多可验证的环境他们可以考虑，但这些环境恰好没有被纳入，因为拥有这方面的能力并不是那么有用。

最大的谜团——有段时间最流行的例子是"strawberry 有几个字母"，模型会弄错这个，这是锯齿状的一个例子。模型现在似乎已经修复了这个。但新的例子是："我想去洗车店洗车，距离 50 米。我应该开车还是走路？"最先进的模型今天会告诉你走路，因为太近了。最先进的 Opus 4.7 怎么可能同时能重构 10 万行的代码库，或者发现零日漏洞，却告诉我要走路去洗车店？这太疯狂了。

只要这些模型仍然是锯齿状的，就表明：第一，可能有些地方不太对劲；第二，你实际上需要保持在 loop 中，你需要把它们当作工具，你需要关注它们在做什么。所以我关于可验证性的所有文章，简而言之，就是试图理解为什么这些东西是锯齿状的。有没有什么模式？我认为这是"可验证"加上"实验室在意"的某种组合。

也许还有一个有启发性的轶事：从 GPT-3.5 到 GPT-4，人们注意到国际象棋水平提高了很多。很多人认为"哦，这只是能力的进步"——但实际上更可能的原因是，大量国际象棋数据进入了预训练集，仅仅因为它出现在数据分布中，模型就比默认情况下提高得多得多。所以 OpenAI 的某个人决定加入这些数据，然后你就有了一个大幅提升的能力。

所以这就是为什么我在强调这个维度：我们在一定程度上受制于实验室在做什么，以及他们碰巧往 mix 里放了什么。你必须去探索他们给你的这个东西，而它没有说明书。它在某些场景下有效，但在其他场景下可能就无效。如果你处在作为 RL 一部分的 circuits 中，你就飞起来了。如果你处于数据分布之外的 circuits 中，你就会很挣扎，你必须弄清楚你的应用处在哪个 circuits 中。如果你不在那些 circuits 里，那么你就需要认真考虑微调，自己做些工作了，因为 LLM 不一定能开箱即用。

### 给创始人的建议

如果你今天是一位创始人，正在考虑创办公司，你试图解决一个你认为可行的问题，一个可验证的领域。但你环顾四周，心想："天哪，实验室在最明显的那些领域——数学、编程等——已经开始达到逃逸速度了。" 你对在座的创始人有什么建议？

我认为在当前范式下，可验证性确实使某个领域变得可行，因为你可以投入大量的 RL。所以一种看法是，即使实验室没有直接关注它，这也仍然成立。所以如果你处于一个可验证的场景中，你可以创建这些 RL 环境或示例，那么这其实为你进行自己的微调创造了条件，你可能会从中受益。这是根本上就有效的技术。如果你有大量多样的数据集和 RL 环境，你可以拉下杠杆——使用你最喜欢的微调框架——然后得到一些实际上效果不错的东西。

我确实认为有一些非常有价值的强化学习环境，人们可以想到，但我认为这些不在实验室的[关注范围]内。我不想在这里剧透，但有一个领域我认为非常有[价值]。抱歉，我不是想在台上故弄玄虚，但确实有一些这样的例子。

反过来看，你觉得哪些东西仍然只是看起来可以自动化？

我确实认为，几乎所有事情最终都可以在某种程度上变得可验证——有些容易些，有些难些。因为即使是写作之类的事情，你也可以想象有一个 LLM 评审团，通过这种方法可能也能得到合理的结果。所以更多的是关于哪些容易哪些难。我认为最终一切都可以自动化。

### Vibe Coding 与 Agentic Engineering

去年你创造了"vibe coding"这个词，而今天我们处在一个感觉更严肃的世界——更多的是 agentic engineering。你认为两者之间的区别是什么？你实际上会怎么称呼我们今天所处的阶段？

是的。我会说 vibe coding 是关于提升每个人在软件领域的下限。下限提升了，每个人都可以 vibe code 任何东西，这很棒，不可思议。但 agentic engineering 是关于保持专业软件之前存在的质量门槛。你不能因为 vibe coding 而引入安全漏洞。你和以前一样对你的软件负责，但你能更快吗？剧透：你可以。但如何正确地做到这一点？所以对我来说，agentic engineering——我这么称呼它，因为我确实认为这是一门工程学科——你拥有这些 agent，它们是尖锐的实体。它们有些容易犯错，有点随机，但它们极其强大。如何协调它们，在不牺牲质量门槛的情况下更快地前进？做好并正确地做到这一点，就是 agentic engineering 的领域。

所以我认为它们是不同的——一个是提升下限，另一个是拓展上限。而我看到的是，agentic engineer 的能力上限非常高。人们过去谈论 10x 工程师——我认为现在这个倍数被放大了很多。10x 并不是你获得的加速。在我看来，那些非常擅长此事的人，上限远不止 10x。

### AI 原生编程是什么样子

Sam Altman 去年在 AI Engineer 大会上说过一句令人难忘的话：不同世代的人以不同方式使用 ChatGPT。如果你三十多岁，你把它当作 Google 搜索的替代品。但如果你十几岁，ChatGPT 是你通往互联网的门户。今天编程中类似的情况是什么？如果我们观察两个人使用 OpenClaw、Claude Code、Codex 编程——一个你认为平庸，另一个你认为完全 AI 原生——你会如何描述这种差异？

我认为就是要尽量利用可用的工具——使用它们的所有功能，投资于你自己的设置。就像以前的工程师都在最大化利用他们使用的工具——无论是 Vim 还是 VS Code，现在则是 Claude Code 或 Codex——投资于你的设置，利用你所能用到的各种工具。我觉得大概就是这样。

一个相关想法——很多人现在可能正在招聘这方面的人才，因为他们想雇佣强大的 agentic engineer。我认为大多数人还没有重构他们的招聘流程以适应 agentic engineer 的能力评估。如果你还在出一些算法题让人解，这仍然是旧范式。我认为招聘必须看起来像这样：给我一个真正的大项目，看一个人如何实现这个大项目。比如让我们为 agent 写一个 Twitter 克隆，把它做得非常好、非常安全，然后让一些 agent 在这个 Twitter 上模拟一些活动，然后我会用 10 个 Codex 5.4x 从 X high 试图攻破你部署的网站，它们会尝试攻破它，而它们不应该能成功。所以可能就是这样的。在这种环境下观察人们构建更大的项目、利用工具的能力，这可能是我主要会看的。

### 品味、判断力与人在回路中

随着 agent 做越来越多的事情，你认为什么人类技能变得更加有价值，而不是更少？

目前答案是，agent 有点像实习生这样的实体。值得注意的是——你基本上仍然需要负责审美、判断力、品味，以及一些监督。

我最喜欢的关于 agent 怪异性的例子是——在 MenuGen 中，你用 Google 账号注册，但用 Stripe 账户购买积分，两者都有邮箱地址。我的 agent 实际上在购买积分时，尝试用 Stripe 的邮箱地址去匹配 Google 邮箱地址——就好像没有一个持久的用户 ID。对人来说，它试图匹配邮箱地址，但你可以为 Stripe 和 Google 使用不同的邮箱地址，然后资金就不会被关联起来。这就是这些 agent 仍然会犯错的地方——你为什么用邮箱地址来交叉关联资金？邮箱地址可以是任意的。你可以用不同的邮箱。这是一个非常奇怪的做法。

所以我认为人们需要负责这个规格说明、这个计划。我实际上甚至不太喜欢 plan mode。显然它非常有用，但我认为这里有更一般的东西：你必须和你的 agent 一起设计一个非常详细的规格说明——可能基本上就是文档——然后让 agent 去写代码，你负责监督和顶层分类，而 agent 做大量底层工作。所以你不必关心一些细节。

另一个例子，关于神经网络中的数组或张量。PyTorch 和 NumPy 以及各种——Pandas 等等——有大量的 API 细节。我已经忘了 keep_dims 还是 keep_dim，是 `dim` 还是 `axis`，是 reshape 还是 permute 还是 transpose。我再也记不住这些东西了。因为你不必记住。这些是由"实习生"处理的细节，因为它们有非常好的记忆力。但你仍然需要知道，例如，有一个底层的 tensor，有一个底层的 view，你可以操作同一存储的 view，或者你可以有不同的存储（那样效率较低）。所以你仍然需要理解这些东西在做什么，以及一些基本原理，这样你就不会不必要地复制内存等等。但 API 的细节现在已经交付出去了。所以你负责品味、工程、设计——以及确保这些是合理的，你在要求正确的事情。比如说，OK，这些必须是唯一的用户 ID，我们要把所有东西关联到它。你在做部分设计和开发，而工程师们在填空。这就是我们目前的状况。

你认为这种品味和判断力是否会随着时间推移变得不那么重要，还是天花板会持续上升？

我希望它会改善。我认为目前它没有改善的原因，再次强调，是因为它不属于 RL 的一部分。可能没有美学方面的成本或奖励，或者这方面的奖励还不够好。我认为当你实际查看代码时，有时我会有点心塞，因为代码并不总是超级棒。非常臃肿，有很多复制粘贴，有脆弱而尴尬的抽象层，它能用，但实在很糟糕。我确实希望这在未来的模型中可以改善。

另一个很好的例子是 microGPT 项目，我试图尽可能简化 LLM 训练。模型们讨厌这个。它们做不到。我不断尝试提示 LLM 让它简化再简化，但它就是做不到。你会感觉你处在 RL circuits 之外。感觉就像拔牙一样。不是光速那样的流畅。所以我确实认为人们仍然负责这个。但并没有根本性的东西在阻止它。只是实验室还没有做到这一点，差不多。

### 动物 vs 鬼魂：理解 LLM 是什么

我很想回到这个锯齿状智能的概念。你写了一篇发人深省的文章，关于动物与鬼魂。大意是我们不是在构建动物，我们是在召唤鬼魂。它们是锯齿状的智能形式，由数据和奖励函数塑造，而不是由内在动机、乐趣、好奇心或赋权——这些都是通过进化产生的。为什么这个框架很重要？它实际上如何改变你构建、部署、评估甚至信任它们的方式？

我写这个是因为我在试图理解这些东西是什么。因为如果你对它们是什么或不是什么有一个好的模型，你就会更擅长使用它们。我不确定这是否真的有实际的力量。我觉得这有点像哲学思考。但这只是在接受一个事实：这些东西不是动物智能。如果你对它们吼叫，它们不会表现得更好或更差——这没有任何影响。这一切都只是统计模拟 circuits，基底是预训练（即统计学），然后 RL 加在上面，所以它在某种程度上增加了"支出"。也许这只是我进入时的心态，或者什么可能有效或什么不太可能有效，或者如何修改它。但我实际上没有"这里有五个改进系统的明显结果"。更多的是对它保持怀疑，并随着时间的推移去摸索。这就是起点。

### Agent 原生基础设施

好的，你深入使用 agent——它们不仅仅是聊天，它们有真正的权限，有本地上下文，它们实际上代表你采取行动。当我们都开始生活在那个世界时，它会是什么样子？

我想这里的很多人可能对 agent 原生的 agentic 环境感到兴奋。一切都需要重写。一切都仍然是为人类编写的，需要被改造。大多数时候，当我使用不同的框架或库之类的东西时，它们的文档仍然是根本上为人类编写的。这是我最喜欢吐槽的点。为什么人们还告诉我该做什么？我什么都不想做。我应该复制粘贴给 agent 的东西是什么？每次有人告诉我"去这个 URL"之类的，我就觉得，啊。

每个人都在兴奋于如何将需要完成的工作负载分解为根本上是世界的传感器和世界的执行器。我们如何让它 agent 原生？基本上首先为 agent 描述它。然后在数据结构上做大量的自动化工作，使数据结构对 LLM 非常可读。

对于 MenuGen，当我写关于 MenuGen 的博客文章时——很多麻烦甚至不是写 MenuGen 的代码，而是把它部署到 Vercel，因为我需要使用所有这些不同的服务，我必须把它们串起来，去它们的设置页面和菜单，配置我的 DNS，这太烦人了。所以这是一个很好的例子：我希望 MenuGen 是这样的——我可以用一个 prompt 给 LLM "构建 MenuGen"，然后我什么都不用碰，它就以同样的方式部署到互联网上了。我认为这是一个很好的测试，看看我们的基础设施是否变得越来越 agent 原生。

最终，我相信我们正在走向一个世界，在那里人和组织都有 agent 代表。"我会让我的 agent 和你的 agent 交谈"来商量我们会议的细节之类的事情。所以我认为这大致就是未来发展的方向。

### 教育：外包思考，而非理解

我想我们得用一个关于教育的问题来结束。因为你可能是世界上最擅长把复杂技术概念讲得简单的人之一，并且对如何围绕它设计教育有深入的思考。当智能变得廉价，我们进入 AI 的下一个时代时，还有什么值得深入学习？

最近有一条推文让我震惊，我几乎每隔一天就会想到它。大意是：**"你可以外包你的思考，但你不能外包你的理解。"**

我认为这句话说得非常好。因为我仍然是系统的一部分，我仍然必须以某种方式——信息仍然需要进入我的大脑——我觉得我成了一个瓶颈，仅仅是知道我们在构建什么、为什么值得做、如何指导我的 agent 等等。所以我确实认为，最终必须有东西来指导思考和加工，而这仍然从根本上受限于理解。

这也是为什么我对所有 LLM 知识库都很兴奋的原因之一——因为我觉得这是我处理信息的一种方式。每当我看到信息的不同投影时，我总觉得获得了洞察。所以对我来说，这实际上就是在某些固定数据上进行合成数据生成的很多 prompt。我非常享受每次阅读一篇文章——我有自己的 wiki，它从这些文章中逐步构建起来，我喜欢对事情提问。我认为这些最终是某种程度上增强理解的工具。而这仍然是一个瓶颈，因为你不能指导——你不能成为一个好的指导者，如果你仍然——因为 LLM 肯定不擅长理解，你仍然是唯一负责理解的人。所以这方面的工具，我认为，是非常有趣和令人兴奋的。

我很期待几年后回到这里，看看我们是否已经被完全自动化出 loop 了，它们是否真的也开始处理理解了。非常感谢你加入我们，Andrej。我们非常感激。
