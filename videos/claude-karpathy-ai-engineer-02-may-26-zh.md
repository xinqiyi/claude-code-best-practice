### 介绍

我们非常激动迎来我们第一位特别嘉宾。他帮助构建了现代 AI，然后解释了现代 AI，偶尔还重新命名了现代 AI。他实际上就在这间办公室里共同创立了 OpenAI。当年在 Tesla 让 Autopilot 真正跑起来的人就是他，他有一种罕见的天赋，让最复杂的技术变革既易于理解又似乎不可避免。你们都因为他去年创造了 vibe coding 这个词而认识他，但就在过去几个月里，他说了更令人震惊的事情：他作为程序员从未感到如此落后。这就是我们今天开始的地方。感谢你，Andrej，加入我们。

是的，你好。很高兴来到这里，为我们开启对话。

### "作为程序员从未感到如此落后"

好的。就在几个月前，你说你作为程序员从未感到如此落后。从你这样的人口中听到这个，非常令人震惊。你能帮我们解读一下吗？这种感觉是令人振奋还是令人不安？

是的，两者都有。首先，像你们中的许多人一样，我已经使用 agentic 工具如 Claude Code 和相关的东西有一段时间了，大概过去一年左右随着它们推出。对于代码块它很擅长，有时会搞砸，你得编辑它们，某种程度上是有帮助的。然后我会说 12 月对我来说是一个清晰的转折点——我当时在休假，所以有更多时间。我想很多其他人也类似——我刚开始注意到，用最新的模型，代码块直接就正确了，然后我不断要求更多，它就是直接正确，我甚至不记得上次纠正它是什么时候了。然后我就越来越信任这个系统，然后我就在 vibe coding 了。

所以这是一个非常鲜明的转变。我认为很多人实际上——我试图在 Twitter 或 X 上强调这一点——因为我认为很多人去年对 AI 的体验是类似 ChatGPT 的东西。但你真的需要重新看看，你需要在 12 月这个时间点重新看看，因为事情已经根本性地改变了——尤其是在这种 agentic 连贯工作流真正开始奏效的方面。所以我会说，正是这种认识让我彻底陷入了无限副项目的兔子洞。我的副项目文件夹极其满，各种随机的项目，一直在 vibe coding。所以这大致是在 12 月发生的，从那以后我一直在观察它的影响。

### Software 3.0：新的计算范式

你谈论了很多关于 LLM 作为一种新计算机的想法。它不仅仅是更好的软件，而是一种全新的计算范式。Software 1.0 是显式规则，software 2.0 是学习到的权重，software 3.0 就是这个。如果这是真的，团队在真正相信这一点的当天，会以什么不同的方式构建东西？

对。所以 software 1.0 是我写代码，software 2.0 是我通过创建数据集和训练神经网络来编程。所以编程有点像排列数据集，也许还有一些目标和神经网络架构。然后发生的是，基本上如果你在一个足够大的任务集上训练这些 GPT 模型或 LLM——隐含地，因为通过在互联网上训练，你必须多任务处理数据集中所有的内容——这些实际上在某种意义上变成了一种可编程计算机。

所以 software 3.0 关于的是，你的编程现在转变为 prompting，而 context window 中的内容是你操控解释器的杠杆，这个解释器就是 LLM，它在数字信息空间中解释你的 context 并执行计算。

我认为有几个例子真正让我明白了这一点。例如当 OpenClaw 推出时，当你想安装 OpenClaw 时，你通常会期望这是一个 bash 脚本，一个 shell 脚本。运行 shell 脚本来安装 OpenClaw。但事实是，为了针对大量不同的平台和不同类型的计算机，你可能运行 OpenClaw，这些 shell 脚本通常会膨胀并变得极其复杂。但问题是你仍然被困在想要编写代码的 software 1.0 宇宙中。实际上，OpenClaw 的安装是一段你应该给 agent 的文本的复制粘贴。所以基本上它是一个小 skill，告诉你"复制粘贴这个并给你的 agent，它会安装 OpenClaw"。这之所以更强大，是因为你现在在 software 3.0 范式中工作，你不需要精确地拼写出设置的所有个别细节。agent 有自己的智能，它将其打包，然后它遵循指令，查看你的环境、你的计算机，并执行智能操作使事情工作，并在循环中调试问题。它就是强大得多。所以这是一种非常不同的思考方式——那段复制粘贴给 agent 的文本是什么？这就是编程范式。

### MenuGen 和"不该存在的 App"

现在，还有一个更极端的例子，当我在构建 MenuGen 时。MenuGen 的想法是，你来到一家餐厅，他们给你菜单。通常没有图片。所以我不知道这些是什么——通常大约 30% 的东西我完全不知道是什么，50%。所以我想拍一张餐厅菜单的照片，并大致显示这些东西可能长什么样。

我 vibe-coded 了这个 app，它基本上让你上传一张照片，它做了所有这些事情，运行在 Vercel 上，它基本上重新渲染菜单，给你所有菜品，并给你一张图片，它使用 image generator 来 OCR 所有不同的标题，用 image generator 获取它们的图片，然后展示给你。

然后我看到了这个的 software 3.0 版本——这让我震惊——它实际上就是：拍你的照片，给 Gemini，说"使用 Nanobanana 把东西叠加到菜单上。"Nanobanana 基本上返回了一张图片，正是我拍的菜单照片，但它实际上把不同的东西渲染到了像素里。这让我震惊，因为实际上我的整个 MenuGen 都是多余的。它在旧范式中工作。那个 app 不应该存在。

Software 3.0 范式更加原始。你的神经网络正在做越来越多的工作，你的 prompt 或 context 只是图片，输出是一张图片，不需要任何 app 在中间。所以人们必须重新框架——不要工作在旧范式中思考什么是已经存在的，也不要把它看作是对现有事物的加速。现在有新的东西可用了。

回到你的编程问题，这甚至不是——我认为这也是用旧思维方式工作的一个例子，因为它不仅仅是关于编程和编程变得更快。这是更一般的信息处理，现在可以自动化了。所以它甚至不仅仅是关于代码。以前的代码处理结构化数据，对吧？你在结构化数据上写代码。但例如我的 LLM 知识库项目——基本上你让 LLM 为你的组织或个人创建 wiki——这甚至不是一个程序。这是以前不可能存在的东西，因为没有代码可以根据一堆事实创建知识库。但现在你可以拿这些文档，基本上以不同的方式重新编译它们，重新排序它们，创建一些新的和有趣的东西作为数据的重新框架。这些是以前不可能做到的新事物。所以我认为这是我不断试图回归的东西：不仅仅是我们可以做什么已经存在且现在更快的事情，而是我认为有新的机会，做以前不可能的事情——我几乎认为这更令人兴奋。

### 推断到 2026 年及以后

我喜欢你描绘的 MenuGen 的演变和二分法。如果你进一步推断，2026 年相当于 90 年代建网站、2010 年代建移动应用、上一个云时代建 SaaS 的是什么？什么会在回顾时看起来完全显而易见，而今天大部分还未被构建？

继续 menu 的例子，很多这些代码不应该存在，大部分工作是神经网络在做。我认为推断看起来会很奇怪，因为你基本上可以想象完全神经的计算机。你输入原始视频——想象一个设备，将原始视频或音频输入基本上是一个神经网络——并使用 diffusion 渲染一个在某种意义上对那一刻独一无二的 UI。

我有点觉得在计算机早期，人们实际上对计算机是像计算器还是像神经网络有些困惑。在 50 年代和 60 年代，哪条路会赢并不明显。当然我们走了计算器的路，最终构建了经典计算，然后神经网络目前是在现有计算机上虚拟化运行的。但我认为很多这种情况会反转，神经网络成为宿主进程，CPU 变成协处理器。所以我们看到了神经网络智能计算的图表将接管并成为 FLOP 的主导消费。所以你可以想象一些非常奇怪和外来的东西，神经网络在做大部分重活。它们使用 tool use 作为某些确定性任务的历史附属物。但真正掌控全局的是这些神经网络。所以你可以想象一些极其外来的东西作为推断，但我认为我们可能会逐步到达那里。

### 可验证性和锯齿状智能

我想谈谈可验证性这个概念——即 AI 将更快、更容易地自动化输出可以被验证的领域。如果这个框架是正确的，什么工作即将比人们意识到的快得多，以及哪些职业人们认为安全但实际上是高度可验证的？

所以我花了一些时间写关于可验证性的内容。基本上，传统计算机可以轻松自动化你能在代码中指定的东西，而这最新一轮的 LLM 可以在某种意义上轻松自动化你可以验证的东西，因为这其中的工作方式是，当前沿实验室训练这些 LLM 时，这些是巨大的 reinforcement learning 环境。所以它们被给予验证奖励，然后由于这些模型的训练方式，它们最终在可验证的领域（如数学、代码和类似的领域）中迅速发展并创造这些锯齿状实体，其能力在这些领域达到顶峰——而在不在这个空间的事情上则停滞不前，有些粗糙。

所以我写关于可验证性的原因是我试图理解为什么这些东西如此锯齿状。其中一些与实验室训练模型的方式有关，但我认为一些也与实验室的关注点以及他们恰好放入数据分布中的内容有关。因为某些东西在经济上显著更有价值，最终创造了更多的环境，因为实验室想要在这些环境中工作。所以我认为代码是一个很好的例子。可能有很多可验证的环境他们可以考虑，但由于拥有这种能力不是那么有用，所以恰好没有被纳入混合。

最大的谜团——一个很长一段时间内最喜欢的例子是"strawberry 有多少个字母"，模型在这方面出了名地出错，这是锯齿状的一个例子。现在模型修补了这个问题，我认为。但新的一个是："我想去洗车，洗车场在 50 米外。我应该开车还是走路？"今天最先进的模型会告诉你走路，因为太近了。怎么可能最先进的 Opus 4.7 同时重构 100,000 行的代码库，或发现 zero-day 漏洞，却告诉我去洗车场走路？这太疯狂了。

只要这些模型保持锯齿状，这表明第一，也许有些地方不太对，或者第二，你真的需要保持在循环中一点，你需要把它们当作工具来对待，你确实需要关注它们在做什么。所以我写的关于可验证性的内容，长话短说，就是试图理解为什么这些东西是锯齿状的。是否有任何模式？我认为这是可验证加上实验室关心的某种组合。

也许还有一个有启发性的轶事：从 GPT-3.5 到 GPT-4，人们注意到国际象棋表现提高了很多，很多人认为"哦，这只是能力的进步"——但实际上是大量国际象棋数据进入了 pre-training 集，仅仅因为它进入了数据分布，模型的进步远远超过了默认情况。所以 OpenAI 的某人决定添加这些数据，现在你就有了一种远远超出预期峰值的能力。

这就是为什么我强调这个维度：我们在某种程度上受制于实验室在做什么，他们恰好放入了什么。你必须实际探索这个没有说明书的东西。它在某些情况下有效，但可能在某些情况下无效。如果你在 RL 的回路内，你就飞起来。如果你在数据分布之外的回路中，你就会挣扎，你必须在你的应用中找出你处于哪些回路中。如果你不在这些回路中，那么你真的需要考虑 fine-tuning 并做一些自己的工作，因为它不一定能从 LLM 开箱即用地出来。

### 给创始人的建议

如果你今天是一个创始人，正在考虑创建一家公司，你试图解决一个你认为可处理的问题，一个可验证的领域，但你环顾四周，你会想，"天哪，实验室在最明显的领域——数学、编码和其他——真的开始达到逃逸速度了。"你对在场的创始人有什么建议？

我确实认为可验证性使某件事情在当前范式中变得可处理，因为你可以向它投入大量的 RL。所以一种看待方式是，即使实验室没有直接关注它，这仍然成立。所以如果你处于一个可验证的环境中，你可以创建这些 RL 环境或示例，那么这实际上为你做自己的 fine-tuning 奠定了基础，你可能从中受益。这是从根本上奏效的技术。如果你有大量多样化的 RL 环境数据集，你可以使用你最喜欢的 fine-tuning framework 并拉动杠杆，得到一些实际上相当不错的东西。

我确实认为有一些非常有价值的 reinforcement learning 环境，人们可以考虑，我认为它们不是[实验室重点]的一部分。我不想直接给出答案，但有一个领域我认为非常有[价值]。抱歉，我不是想在台上含糊其辞，但这方面是有例子的。

另一方面，你觉得什么在远处看起来可以自动化，但实际上不是？

我确实认为最终几乎一切都可以在某种程度上变得可验证——有些东西比其他的更容易。因为即使是像写作之类的事情，你可以想象有一个 LLM 裁判委员会，可能通过这种方法得到一些合理的东西。所以这更多是关于什么容易或困难。我认为最终一切都是可自动化的。

### Vibe Coding vs Agentic Engineering

去年你创造了 vibe coding 这个词，今天我们处于一个感觉更严肃的世界——更多是 agentic engineering。你认为两者之间的区别是什么，你会把我们现在所处的阶段叫什么？

是的。我会说 vibe coding 是关于提高每个人在软件方面的下限。所以下限提高了，每个人都可以 vibe code 任何东西，这很棒，难以置信。但我会说 agentic engineering 是关于保持专业软件中以前存在的质量门槛。所以你不能因为 vibe coding 而引入漏洞。你仍然像以前一样对你的软件负责，但你能更快吗？剧透是：你能。但如何正确地做到这点？所以对我来说 agentic engineering——当我这么称呼它时，因为我确实认为它是一门工程学科——你有这些 agent，它们是这些尖刺状的实体。它们有点容易出错，有点随机，但它们非常强大。你如何协调它们以更快地前进而不牺牲质量门槛？做好并正确地做到这一点是 agentic engineering 的领域。

所以我把它们看作是不同的——一个是关于提高下限，另一个是关于向上推断。而我所看到的是，agentic engineer 能力的上限非常高。人们以前谈论 10x 工程师——我认为这被放大了很多。10x 不是你获得的速度提升。在我看来，非常擅长这个的人目前从我的角度看远不止 10x。

### AI-Native 编码是什么样的

Sam Altman 去年在 AI Engineer 上说的一件事令人难忘：不同年代的人使用 ChatGPT 的方式不同。如果你 30 多岁，你把它当作 Google 搜索的替代品。但如果你是青少年，ChatGPT 是你进入互联网的门户。今天编码中的平行现象是什么？如果我们观察两个人使用 OpenClaw、Claude Code、Codex 编码——一个你会认为水平平庸，一个你会认为完全是 AI-native——你会如何描述差异？

我认为就是试图从可用工具中获得最大的收益——利用它们所有的功能，投资于你自己的设置。所以就像以前所有工程师都习惯从他们使用的工具中获得最大收益——无论是 Vim 还是 VS Code 还是现在的 Claude Code 或 Codex——投资于你的设置并利用大量可用的工具。我认为它大致就是这个样子。

一个相关的想法——很多人可能现在正在为此招聘，因为他们想雇用强大的 agentic engineer。我确实认为大多数人还没有重构他们的招聘流程以适应 agentic engineer 能力。如果你给出需要解决的谜题，这仍然是旧范式。我会说招聘应该看起来像：给我一个真正大的项目，看某人实现那个大项目。比如让我们为 agent 写一个 Twitter 克隆，然后真的让它很好，让它真的安全，然后让一些 agent 在这个 Twitter 上模拟一些活动，然后我将使用 10 个 Codex 5.4x 以 X high 来尝试破坏你部署的网站，它们应该不能破坏它。所以也许看起来像这样。在这种环境下观察人们，构建更大的项目并利用工具，可能是我会主要关注的。

### 品味、判断和 Human in the Loop

随着 agent 做得更多，什么人类技能变得更有价值而不是更少？

现在答案是 agent 有点像这些实习生实体。这很了不起——你基本上仍然必须负责美学、判断、品味和一点监督。

我最喜欢的 agent 奇怪之处的例子之一是——对于 MenuGen，你用 Google 账户注册，但你用 Stripe 账户购买积分，两者都有电子邮件地址。我的 agent 实际上尝试过，当你购买积分时，它使用 Stripe 的电子邮件地址分配给 Google 电子邮件地址——好像没有一个持久的用户 ID。对于人来说，它试图匹配电子邮件地址，但你可以对你的 Stripe 和 Google 使用不同的电子邮件地址，基本上不会关联资金。所以这就是这些 agent 仍然会犯的错误——比如为什么要使用电子邮件地址来尝试交叉关联资金？它们可以是任意的。你可以使用不同的电子邮件。这是一个非常奇怪的事情。

所以我认为人们必须负责这个 spec，这个计划。我实际上甚至不喜欢 plan mode。显然它非常有用，但我认为这里有更一般的东西，你必须与你的 agent 合作设计一个非常详细的 spec——可能基本上是文档——然后让 agent 编写它们，你负责监督和顶层类别，但 agent 在幕后做很多工作。所以你不关心一些细节。

另一个例子是神经网络中的数组或 tensor。在 PyTorch 和 NumPy 之间以及所有不同的 API 细节——Pandas 等等之间，有很多细节。我已经忘记了 keep_dims 与 keep_dim，或者它是 `dim` 还是 `axis`，或者 reshape、permute 还是 transpose。我不再记得这些了。因为你不需要。这些是实习生处理的细节，因为他们有非常好的记忆力。但你仍然必须知道，例如，有一个底层的 tensor，有一个底层的 view，然后你可以操作相同存储的 view，或者你可以有不同的存储，这将更低效。所以你仍然必须理解这些东西在做什么以及一些基本原理，这样你就不会不必要地复制内存等等。但 API 的细节现在已经交给别人了。所以你负责品味、工程、设计——以及它有意义，你在要求正确的东西。好的，这些必须是唯一的用户 ID，我们将把所有东西都绑定到它们。所以你在做一些设计和开发工作，而工程师们在做填空题。这就是我们目前的位置。

你认为这种品味和判断越来越不重要的可能性有多大，还是上限会不断提高？

我希望它会改进。我认为现在它没有改进的原因可能是，再次强调，它不在 RL 的一部分。可能没有美学成本或奖励，或者它不够好或者类似的东西。我确实认为当你实际看代码时，有时我有点心惊，因为代码不一定总是超级棒的。它非常臃肿，有很多复制粘贴，有尴尬的抽象，脆弱但它能工作但就是真的很恶心。我确实希望这在未来的模型中能改进。

一个很好的例子也是这个 microGPT 项目，我试图将 LLM 训练简化到尽可能简单。模型讨厌这个。它们做不到。我试图一直 prompting LLM 更多地简化，更多地简化，它就是做不到。你感觉你不在 RL 回路里。感觉像是在拔牙。不像光速。所以我确实认为人们仍然负责这方面。但没有什么根本性的东西在阻止它。只是实验室还没有做到，几乎如此。

### Animals vs Ghosts：理解 LLM 是什么

我想回到锯齿状智能形式的这个想法。你写了一篇发人深省的文章关于 animals versus ghosts。这个想法是，我们不是在建造动物，我们在召唤幽灵。这些是被数据和 reward function 塑造的锯齿状智能形式，但不是由内在动力、乐趣、好奇心或 empowerment 塑造的——这些是通过进化产生的。为什么这个框架很重要，它实际上如何改变你构建、部署、评估甚至信任它们的方式？

我之所以写这个是因为我试图理解这些东西是什么。因为如果你对它们是什么或不是有一个好的模型，那么你在使用它们时就会更有能力。我不确定它是否有实际力量。我认为这有点哲学化。但这是接受这样一个事实：这些东西不是动物智能。如果你对它们大喊大叫，它们不会工作得更好或更差——没有任何影响。全都是统计模拟回路，其中 substrate 是 pre-training（所以是统计学），然后 RL 被加在上面，所以它有点增加了附属物。也许这只是一种心态，关于我正在进入什么，或者什么可能有效或无效，或者如何修改它。但我实际上没有，这里是让你的系统更好的五个明显结果。更多是对它持怀疑态度，并随着时间的推移去弄清楚。这就是开始的地方。

### Agent-Native Infrastructure

好的，你如此深入地与那些不仅仅聊天而是有真正权限、有本地 context、实际上代表你采取行动的 agent 合作。当我们都开始生活在那个世界中时，世界会是什么样子？

我认为这里的很多人可能对 agent-native agentic 环境的样子感到兴奋。一切都必须重写。一切仍然根本上为人类编写，必须被移动。我大部分时间使用不同的 framework、library 或类似的东西时，它们仍然有根本上为人类编写的文档。这是我最喜欢的 pet peeve。为什么人们还在告诉我该做什么？我不想做任何事情。我应该复制粘贴给我的 agent 的东西是什么？每次我被告诉"去这个 URL"或类似的东西，就是啊。

每个人都对如何将需要发生的工作负载分解为根本上对世界的传感器和世界的执行器感到兴奋。我们如何使其 agent-native？基本上首先向 agent 描述它。然后围绕对 LLM 非常可读的数据结构有很多自动化。

对于 MenuGen，有名的是当我写关于 MenuGen 的博客文章时——很多麻烦甚至不是为 MenuGen 编写代码，而是在 Vercel 中部署它，因为我必须与所有这些不同的服务合作，我必须把它们串起来，我必须去它们的设置和菜单，配置我的 DNS，这太烦人了。所以这是一个很好的例子：我希望 MenuGen，我能够给 LLM 一个 prompt "build MenuGen"，然后我不用碰任何东西，它就部署在互联网上。我认为这将是对我们多少基础设施正在变得更加 agent-native 的一个很好的测试。

最终我确实认为我们正走向一个世界，在那里有 agent representation 为人和组织。 "我让我的 agent 和你的 agent 谈"来弄清楚我们会议的一些细节或类似的事情。所以我确实认为这大致上是事情发展的方向。

### 教育：外包思考，但不外包理解

我认为我们必须在教育问题上结束。因为你可能是世界上在使复杂技术概念变得简单方面最出色的人之一，并且对我们如何围绕它设计教育有深刻的思考。当智能变得廉价时，随着我们进入 AI 的下一个时代，什么仍然值得深入学习？

最近有一条推文让我震惊，我每隔一天都在想它。内容大致是：**"你可以外包你的思考，但你不能外包你的理解。"**

我认为这说得非常好。因为我仍然是系统的一部分，我仍然必须以某种方式——信息仍然必须以某种方式进入我的大脑——我感觉我正在成为瓶颈，甚至仅仅是知道我们试图构建什么、为什么值得做、如何指导我的 agent 等等。所以我确实仍然认为最终必须有什么东西来指导思考和加工，而这仍然在根本上受到理解的约束。

这也是我对所有 LLM 知识库非常兴奋的原因之一——因为我觉得这对我来说是一种处理信息的方式。每当我看到信息的不同投影时，我总是觉得自己获得了洞察。所以这实际上对我而言是一系列在固定数据上进行 synthetic data generation 的 prompts。我非常喜欢当我读到一篇文章时——我有我的 wiki，从这些文章中建立起来，我喜欢问关于事情的问题。我认为最终这些是以某种方式增强理解的工具。而这仍然是一个瓶颈，因为然后你不能指导，你如果仍然——因为 LLM 肯定不擅长理解，你仍然在这方面独特地负责。所以用于此目的的工具，我认为是非常有趣和令人兴奋的。

我很兴奋几年后能回到这里，看看我们是否已经完全被自动化挤出循环，它们实际上也接管了理解。非常感谢你加入我们，Andrej。我们真的很感激。
