[`0:01`](https://youtu.be/julbw1JuAz0?t=1) 你是第一个写 O'Reilly 出版的 TypeScript 书籍的人。

[`0:04`](https://youtu.be/julbw1JuAz0?t=4) 是的，我在日本的一个小镇上发现了那本书的日文译本。那真是一个非常酷的时刻。然后我意识到我完全不记得 TypeScript 了。现在，我们到了这样一个阶段：Claude Code 写了 Anthropic 平均大约 80% 的代码。我每天大概写 10 到 20 个 PR 请求。Opus 4.5 和 Claude Code 写了每一个 PR 的 100% 代码。我没有手动编辑过一行代码。

[`0:22`](https://youtu.be/julbw1JuAz0?t=22) Andrej Karpathy 发帖说他从未像现在这样感觉自己作为程序员如此落后。

[`0:26`](https://youtu.be/julbw1JuAz0?t=26) 这是我确实在努力应对的事情。模型的进步如此之快，以至于在旧模型上有效的想法可能在新模型上不再有效。我对这个时代的一个比喻是 15 世纪的印刷机，因为那时有一群知道如何书写的抄写员。

[`0:42`](https://youtu.be/julbw1JuAz0?t=42) 一些国王是文盲，但他们雇佣着抄写员。

[`0:45`](https://youtu.be/julbw1JuAz0?t=45) 如果你想想那些抄写员的命运，他们不再是抄写员了，但现在出现了一个新的类别——作家和作者。这些人现在存在了。他们存在的原因是文学市场在扩大。当你加入世界顶级 AI 实验室之一，你的第一个 PR 被拒绝了——不是因为代码写得不好，而是因为你是手写的。这正是 Boris Cherny 加入 Anthropic 时发生的事情。Boris 是 Claude Code 的创建者和工程负责人。在加入 Anthropic 之前，他在 Meta 工作了 7 年，负责 Instagram、Facebook、WhatsApp 和 Messenger 的代码质量，并且是公司最多产的代码作者和代码评审者之一。在今天的节目中，我们将讨论 Claude Code 如何从一个副项目变成增长最快的开发者工具之一，以及 Anthropic 内部关于是否要发布它的辩论。Boris 的日常工作流程——每天提交 20 到 30 个 PR 请求，零手写代码——以及在 AI 编写所有代码的时代，代码评审如何运作。为什么 Boris 相信我们正在经历一个像印刷机一样具有变革性的时代，以及哪些工程技能现在更重要，哪些不再那么重要。如果你想了解最接近 AI 编程助手的人之一如今是如何实际构建软件的，以及这对我们其他工程师意味着什么，这期节目就是为你准备的。本期节目由 Statsig 呈现，这是一个集标志、分析、实验等功能于一体的统一平台。查看节目介绍，了解更多关于他们以及我们的其他季度赞助商 Sonar 和 Work OS 的信息。你是如何进入科技行业、软件工程以及编程这个领域的？

[`2:14`](https://youtu.be/julbw1JuAz0?t=134) 这要追溯到很久以前。我觉得当时有两条平行路径交叉了。在我大概 13 岁的时候，我开始在 eBay 上卖我的旧宝可梦卡。我发现在 eBay 上，你实际上可以写 HTML。我在看别人的宝可梦卡列表时发现，有些人有大的彩色字体之类的东西。然后我发现了 `blink` 标签，我用了它。

[`2:41`](https://youtu.julbw1JuAz0?t=161) 如果我用上 `blink` 标签，我就能把卡卖到 99 美分而不是 49 美分。我通过这种方式了解了 HTML。然后我买了一本 HTML 书，进一步学习了 HTML。第二件事是，大概也是在中学时期。我们有那种老式的 TI83 图形计算器，用来上数学课。我意识到，如果我把数学考试的答案直接编进计算器里，我就能考得更好。于是我写了小程序来编程输入答案。后来考试变难了，我不得不编写求解器程序而不是预存答案，因为我事先不知道系数之类的值。再后来数学更难了，比如第二年，我不得不从 BASIC 降到汇编，只为了让程序跑得快一点。

[`3:28`](https://youtu.be/julbw1JuAz0?t=208) 哦，哇。所以你在高中就开始用汇编了。

[`3:30`](https://youtu.be/julbw1JuAz0?t=210) 我想应该是中学或高中，大概八年级或九年级的样子。然后我意识到班上的同学开始注意到我有求解器，他们有点嫉妒了，所以我买了一条串口线，这样也可以给他们用。然后下一次数学考试，全班都拿了 A。老师很疑惑，"怎么回事？"最后她发现了，说："好吧，你们蒙混过关了一次，但别再这样了。"但对我来说，这非常实用。在学校我学的是经济学。实际上我辍学去创业了，我从没想过编程会成为职业。对我来说它一直很实用。编程是构建东西、创造有用东西的手段。第一家创业公司是我和朋友们想弄到大麻，于是做了一个大麻评测网站。我们做了一个网站，联系了不同的药房，想弄到一些大麻样品来帮他们写评测。结果它居然火了。然后我对此更感兴趣了，当时没人对这些东西做检测，所以我开始接触化学检测和分析。之后我又做了好几家创业公司，后来很早就加入了 YC，我是 Palo Alto 一家 YC 创业公司的第一个员工。

[`4:54`](https://youtu.be/julbw1JuAz0?t=294) 你是如何决定一家接一家地做创业公司的？

[`4:57`](https://youtu.be/julbw1JuAz0?t=297) 靠感觉吧，因为创业从来不是一条直线。你总在不断转型。你必须弄清楚市场想要什么，用户想要什么。你最初的想法总是错的。你总是先尝试一个东西，但那个想法只是一个假设，几乎总是要转型一次、两次、三次。在一家叫 Agile Diagnosis 的医疗软件公司，当时是 YC 的早期项目，大概是 2011、2012 年左右。那是为医生做的医疗软件。我们的想法是，每家医院的临床决策方案差异很大，芝加哥有一家医院的心脏病症状方案特别好。我们想，如果全美所有医院都用同一个方案，治疗效果会不会很好？所以我们试图标准化它，做了一个供医生使用的决策树软件。我写了其中一些代码。团队很小，只有几个人。我把软件做在网页浏览器里。我记得那还是在 Internet Explorer 6 的时代，医院用的就是那个。

[`6:06`](https://youtu.be/julbw1JuAz0?t=366) 我写了一个 SVG 渲染器，因为它是一个可视化的决策树。我们发布了它，然后 DAU 图表是平的。我们当时在 Palo Alto，和包括 UCSF 在内的几家医院进行试点。我当时骑摩托车，于是骑到 UCSF，跟诊了医生几天，看看他们实际如何使用。

[`6:42`](https://youtu.be/julbw1JuAz0?t=402) 我意识到医生其实没有时间坐下来用电脑，因为他们看完一个病人，可能只有 5 分钟时间准备下一个病人。在这 5 分钟里，他们要走过走廊，走到工作站，打开一台完全老旧的电脑。等它启动就要 3 分钟。然后打开 IE 6，又要 30 秒。再打开我们做的应用，登录进去，5 分钟就没了，他们根本没时间用。于是我们把所有东西重写成 Android 版本，但他们还是不用。我们发现医生身后总跟着一群住院医生。在这种情况下，社交因素很重要，对吧？重要的是他们被视为权威。他们不想让别人看到自己在用手机。于是我们又转型了。那时我们想，也许医生不是目标用户。也许我们应该面向护士或 X 光技师。那时我离开了，因为我觉得这已经偏离了我真正想做的事情。对我来说，最有趣的事情就是找到产品市场契合点，因为它总是出人意料。你不能只有一个宏大的想法，因为那个想法很可能是错的。所以你提出假设，沿着它往下走，看看到底对不对。我觉得你讲这个故事非常有趣，因为很多创业成功故事里，我们听到的都是成功之路。但首先，很多创业公司都是这样的。其次，让我印象深刻的是，你当时是作为软件工程师被招进去的，对吧？那还是在"产品工程师"这个词出现之前。但你直接骑上摩托车，去现场跟诊，观察他们如何使用、为什么不使用。获取想法。我觉得，这就是当年和现在优秀软件工程师的特质，对吧？你并不专注于技术本身，而是专注于结果。

[`8:31`](https://youtu.be/julbw1JuAz0?t=511) 是的。工程师有不同的类型，也有不同的做事方式。即便在我们现在的团队里，我看着 Jared Sumar 这样的工程师，他有着惊人的技术头脑，比任何人都更懂系统。你需要这样的人，需要这种深度的人。对我来说，工程一直都是实用性的。我一直是个通才，不论是做设计、工程还是用户研究。AI 投资软件工程的逻辑很直接：随着 AI 写更多代码，需要验证的代码也越多。但有一个问题：AI 生成的代码平均比人类写的代码更难验证。这就是 Sonar（SonarQube 的开发者）存在的原因。作为 AI 赋能世界的关键验证层，Sonar 确保 AI 带来的速度和数量不会损害你的代码库。Sonar 的竞争地位建立在 17 年的专业经验之上，这是任何基础模型都无法复制的。我们说的是深度分析引擎，比如符号执行和跨仓库数据流追踪，它们模拟代码的实际行为，而不仅仅是代码的表面内容。为弥合 AI 生产力与代码质量之间的鸿沟，Sonar 发布了 SonarQube MCP server。这个工具充当 AI 应用与 SonarQube 平台之间的通用翻译器。通过使用 Model Context Protocol，它赋予 Claude Code、GitHub Copilot 和 Cursor 等 AI 工具直接访问 SonarQube 分析能力的能力。无需切换上下文，你的 AI agent 就变成了一个全功能的代码评审和质量保证助手，能够分析代码片段的问题、按严重性过滤 bug，甚至在提交代码前检查项目的质量门状态。无论你是在使用编程助手还是扩展到完整的 agentic workflow，Sonar 都提供了《财富》100 强中 75% 的公司所依赖的自动化验证。让开发者拥有创新的自由，而不必担心破坏代码库。访问 sonarsource.com/pragmatic 了解更多关于 Sonar 如何让你以 AI 的速度自信开发。接下来，让我们回到 Boris 的职业生涯以及他在创业公司学到的东西。我的第一份工作是在 16 岁，当时我只想买一把电吉他。于是我开始做自由职业。我心想："好吧，那我就做网站。"当时 Fiverr 可能还没出现。有一些其他的自由职业网站。我建了个网站，开始投标。第一笔薪水全部用来买了一把电吉他。这非常实用，对吧？因为在这种模式下，你得自己做工程、做账、做设计、跟客户沟通。对我来说一直都是这样。经过几家创业公司之后，你去了 Facebook，也就是现在的 Meta。你在那里工作了七年。能跟我们讲讲你在那里的工作和学到的经验吗？你在七年的时间里获得了四次晋升，职业发展非常出色。从那段经历中你带走了什么？

[`11:39`](https://youtu.be/julbw1JuAz0?t=699) 好的。我从 Facebook Groups 开始。那是我第一次在 Vlad Klesnikov 手下工作。他好像现在还在 Facebook，在别的团队。那真的很酷。当时和我一起工作的很多人都是早期的 JavaScript 开发者。我做了很多 JavaScript 相关的工作。有趣的是，我后来一直和这些人有交集。Vlad 之前做了 Bolt.js，那是后来广告管理器的框架，最终变成了 ReactJS。我不断与这些人重逢，后来还有更多这样的人。总之，我在 Facebook Groups 工作。我对它非常兴奋，因为它有连接人们与社区的使命。这吸引了我。当时我是 Reddit 的重度用户。我十几岁就开始用 Reddit，因为我不认识其他会编程的人。即使在大学里，我也不怎么认识会编程的人。

[`12:37`](https://youtu.be/julbw1JuAz0?t=757) 说实话，我一直对这件事有点不好意思，因为我觉得这是很书呆子气的事情。这是我会做的事情，但我想做个酷孩子，我不能告诉别人我会编程，因为太书呆子了。后来有一天，我在 Reddit 上发现了一个编程社区，我震惊了：居然还有其他人也喜欢这个？这是多么奇怪的爱好，多么小众。能遇到志同道合的人、建立这种联系，让我非常兴奋。所以我想做这件事，想以某种方式为此做贡献。我在 Facebook Groups 工作了一段时间。然后做了很多不同的项目。后来我成为了 Facebook Groups 的技术负责人，慢慢成长。组织在扩大，工作内容也变了。从构建变成写文档、协调工作、委派任务。那时文化也在变化。早期的 Facebook 文化在消失。文档越来越多，对齐会议越来越多。还有很多基础性的工作，比如隐私、安全等等。早期为了增长走了很多捷径，但到了一定时候你必须偿还这笔债。那正是那个时候。后来我在 Instagram 工作了几年。这也有个有趣的故事。我妻子拿到了一份工作 offer，她非常兴奋。她来问我："嘿，我拿到了这个 offer，但我们要搬家，可以吗？"我说："没问题，我在科技行业工作，可以远程。"她说工作在日本。我问："日本哪里？"是日本的乡下。

[`14:25`](https://youtu.be/julbw1JuAz0?t=865) 我试图找一个能赞助我的团队，因为公司有一些关于时区和团队办公地点的奇怪 HR 规定。Instagram 在东京有一个小团队，Will Bailey 在带这个团队，他也是 Instagram Stories 的创始人。他做了我一段时间的经理。我们一起发展那个团队。我在乡下远程工作，团队大部分人在东京。在那段时间，我开始在 Instagram 上做一些 hack，那个技术栈简直太疯狂了。Facebook 拥有世界上最好的 Web 服务栈。从 Hack 语言到 HHVM 运行时，到 GraphQL 传输层，再到 Relay 这样的客户端库——所有东西都是高度优化的，还有 React。那太棒了，世界上没有其他技术栈能做到这么好。然后我去了 Instagram，它的技术栈是 Python，类型检查器不工作，点击跳转到定义也不工作，是一个拼凑起来的 Django，加上一个……Syon 运行时？基本上什么都不好用。我加入 Instagram 的实验室团队，目标是为 Instagram 找到下一个大项目。我们尝试了一些东西，但我很快意识到，在这个技术栈上我根本没有效率，因为它实在太差了。于是我开始做开发者基础设施，因为我们需要修复它。我们做了几个项目。一个是把代码从 Python 迁移到 Facebook 的大型单体库。另一个是从 REST 迁移到 GraphQL。这些项目实际上还在进行中，需要几百名工程师花很多年才能完成。代码库很大，迁移规模也很大。不过现在已经快多了。

[`16:21`](https://youtu.be/julbw1JuAz0?t=981) 是的，这是完美的使用场景。然后我在这方面越做越深入。到离开 Instagram 时，我一直在做开发者基础设施，领导着这些迁移项目。也是在那里，我遇到了 Fiona Fun，她现在管理着 Claude Code 团队。我和她合作过，她是一个非常出色的领导者，有着深厚的技术功底和历史经验。我觉得没有比她更好的经理了。然后我也开始做代码质量方面的工作。在 Instagram 的工作范围逐渐扩大。到我离开时，我是整个 Meta 的代码质量负责人。我负责 Instagram、Facebook、Messenger、WhatsApp、Reality Labs 等所有代码库的质量。在 Meta，有一个叫 Better Engineering 的项目。我记得大概是 2016 年或 2018 年，Zuck 要求公司每位工程师将 20% 的时间用于修复技术债务。

[`17:17`](https://youtu.be/julbw1JuAz0?t=1037) 哦，有意思。

[`17:19`](https://youtu.be/julbw1JuAz0?t=1039) 我们称之为 Better Engineering。

[`17:22`](https://youtu.be/julbw1JuAz0?t=1042) 其中一部分是自下而上的，团队最清楚他们需要修复哪些技术债务；另一部分是自上而下的，比如需要做大型迁移，迁移到新的语言特性、新的框架等等。在 Facebook 的规模下，每年有几万个这样的迁移。我领导了所有这些工作，并且很快意识到它需要更多的秩序。没有目标，没人知道结果是什么，也没有追踪。于是我们开发了一系列方法。一个想法是建立一个集中化的优先级排序机制，用来排定不同代码质量工作的优先级。第二件事是衡量代码质量对工程生产力的影响，结果发现影响非常显著。
