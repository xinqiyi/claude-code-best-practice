## 用户输入

```text
$ARGUMENTS
```

你**必须**解析用户输入，提取功能标识（`rpi/` 中的文件夹名称）。

## 目的

该命令为功能请求创建全面的规划文档。它在功能的 RPI 文件夹中生成详细规格说明、技术设计和实施计划。

**前提条件**：
- 功能文件夹存在于 `rpi/{feature-slug}/`
- 研究已完成并带有 GO 推荐（`rpi/{feature-slug}/research/RESEARCH.md` 存在）

**输出位置**：所有文件保存到 `rpi/{feature-slug}/plan/`

**这是 RPI 工作流程的第 3 步**（研究批准 GO 之后）。

## 概览

1. **加载上下文**：阅读研究报告和项目章程（如存在）
2. **理解需求**：解析功能范围和要求
3. **分析技术需求**：审查架构和依赖关系
4. **设计架构**：创建高层架构和 API 合约
5. **分解实施**：创建分阶段任务分解
6. **生成文档**：创建结构化文档文件
7. **验证输出**：确保所有质量门禁通过
8. **报告完成**：提供总结和后续步骤

## 阶段

### 阶段 0：加载上下文

**前提条件**：功能标识已提供

**流程**：
1. **验证研究已完成**：
   - 检查 `rpi/{feature-slug}/research/RESEARCH.md` 是否存在
   - 验证 GO 推荐（如果是 NO-GO 或 CONDITIONAL 则发出警告）

2. **阅读研究发现**：
   - 提取产品分析
   - 提取技术发现
   - 提取技术可行性评估
   - 记录风险和约束

3. **加载项目章程**（如存在）：
   - 在仓库中查找章程或原则文档
   - 提取相关约束和偏好

**输出**：
- 研究总结
- 章程上下文（如找到）
- 规划约束

**验证**：
- [ ] 研究报告存在
- [ ] GO 推荐已确认
- [ ] 章程已加载（如存在）

---

### 阶段 1：理解功能需求

**前提条件**：阶段 0 完成

**流程**：
1. **从研究报告中解析功能描述**：
   - 提取功能名称和主要目标
   - 识别目标组件
   - 理解面向用户的功能与技术功能
   - 确定功能复杂程度

2. **识别受影响的组件**：
   - 主要组件（功能所在位置）
   - 次要组件（集成点）
   - 所需的共享工具
   - 外部依赖

3. **研究现有模式**：
   - 在代码库中搜索类似功能
   - 审查组件架构和模式
   - 识别可复用代码和模式

**输出**：
- 功能范围文档（内部）
- 受影响组件列表
- 现有模式目录

**验证**：
- [ ] 功能名称和目标已明确定义
- [ ] 目标组件已识别
- [ ] 功能复杂度已评估

---

### 阶段 2：分析技术需求

**前提条件**：阶段 1 完成

**流程**：
1. **审查组件架构**：
   - 阅读组件 README 和文档
   - 审查现有代码结构
   - 识别所使用的架构模式

2. **识别技术依赖**：
   - 内部依赖（其他组件、共享工具）
   - 外部依赖（API、服务、库）
   - 数据库/存储需求
   - 认证/授权需求

3. **评估集成点**：
   - 需要创建或修改的 API
   - 所需的数据库 schema 变更
   - 事件/消息流
   - 前端-后端集成

4. **评估技术风险**：
   - 对现有功能的破坏性变更
   - 性能影响
   - 安全问题
   - 数据迁移需求

**输出**：
- 技术需求文档（内部）
- 依赖关系图
- 集成点图表
- 风险评估

**验证**：
- [ ] 组件架构已理解
- [ ] 所有依赖已识别
- [ ] 集成点已映射
- [ ] 技术风险已评估

---

### 阶段 3：设计功能架构

**前提条件**：阶段 1-2 完成

**Agent**：senior-software-engineer

**流程**：
1. **设计高层架构**：
   - 组件/模块结构
   - 数据流图
   - API 接口
   - 数据库 schema 变更

2. **定义实施方法**：
   - 文件结构和组织
   - 代码组织模式
   - 测试策略
   - 错误处理方法

3. **规划数据库/存储变更**（如适用）：
   - 新集合/表
   - Schema 修改
   - 迁移策略
   - 数据验证规则

4. **设计 API 合约**（如适用）：
   - 请求/响应格式
   - 认证需求
   - 错误响应

5. **规划测试策略**：
   - 单元测试需求
   - 集成测试场景
   - 端到端测试用例

**输出**：
- 架构设计文档（内部）
- API 规格说明
- 数据库 schema 设计
- 测试策略

**验证**：
- [ ] 高层架构已设计
- [ ] 实施方法已定义
- [ ] 数据库变更已规划（如需）
- [ ] API 合约已指定（如需）
- [ ] 测试策略已完成

---

### 阶段 4：分解实施任务

**前提条件**：阶段 1-3 完成

**流程**：
1. **识别实施阶段**：
   - 将功能分解为 3-5 个逻辑阶段
   - 每个阶段应交付可工作、可测试的功能
   - 阶段之间应逐步递进

2. **为每个阶段创建任务分解**：
   - 列出具体的实施任务
   - 估算复杂度（低/中/高）
   - 识别任务依赖
   - 分配到适当的代码区域

3. **定义成功标准**：
   - 每个阶段的验收标准
   - 测试需求
   - 文档需求

4. **识别并行化机会**：
   - 可以同时进行的任务
   - 前端/后端并行工作
   - 独立模块开发

**输出**：
- 分阶段实施计划
- 带估算的任务分解
- 每个阶段的成功标准
- 依赖关系图

**验证**：
- [ ] 功能分解为 3-5 个逻辑阶段
- [ ] 每个阶段有具体任务
- [ ] 所有任务有复杂度估算
- [ ] 依赖关系已明确标记
- [ ] 成功标准已定义

---

### 阶段 5：生成文档

**前提条件**：阶段 1-4 完成

**Agent**：documentation-analyst-writer（通过 Task 工具）

**流程**：
1. **生成 pm.md**（产品需求）：
   - 功能描述和用户故事
   - 章程对齐（如适用）
   - 商业价值和成功指标
   - 用户角色和用例
   - 验收标准
   - 范围外事项

2. **生成 ux.md**（用户体验设计）：
   - 用户界面原型（文本描述）
   - 用户流程和交互
   - 可访问性考虑
   - 错误状态和边界情况

3. **生成 eng.md**（技术规格说明）：
   - 架构设计
   - API 规格说明
   - 数据库 schema 变更
   - 技术栈
   - 技术风险及缓解措施

4. **生成 PLAN.md**（实施路线图）：
   - 分阶段实施分解
   - 每个阶段的任务列表及估算
   - 依赖关系和顺序
   - 每个阶段的成功标准
   - 测试需求
   - 验证检查点

**输出文件**（全部保存到 `rpi/{feature-slug}/plan/`）：
- `pm.md` - 产品需求
- `ux.md` - 用户体验设计
- `eng.md` - 技术规格说明
- `PLAN.md` - 详细实施路线图

**验证**：
- [ ] 全部 4 个文件存在（pm, ux, eng, PLAN）
- [ ] pm.md 涵盖业务需求
- [ ] ux.md 涉及用户体验
- [ ] eng.md 提供技术规格说明
- [ ] PLAN.md 包含分阶段实施
- [ ] 无占位符文本残留
- [ ] Markdown 格式整洁

### Agent 调用

**自定义 Agent**（product-manager、senior-software-engineer、ux-designer）：
- Claude Code 会自动从 `.claude/agents/` 检测这些 agent
- 通过自然语言引用："作为 senior-software-engineer agent 行事..."
- 无需 Task 工具调用

**内置 Agent**（documentation-analyst-writer）：
- 使用 Task 工具，设置 `subagent_type="documentation-analyst-writer"`

## 错误处理

**如果研究报告不存在**：
- 操作：停止并通知用户
- 消息："未找到研究报告。请先运行 `/rpi:research`。"

**如果研究推荐为 NO-GO**：
- 操作：警告用户但允许继续
- 消息："研究推荐 NO-GO。是否仍然继续？（y/n）"

**如果目标组件不存在**：
- 操作：向用户确认是否为新组件
- 消息："未找到组件。这是一个新组件吗？"

**如果文档 agent 失败**：
- 操作：直接生成文档
- 警告："文档可能不完全符合标准"

## 说明

- **前提条件**：研究已完成并带有 GO 推荐
- **属于 RPI 工作流程**：第 3 步，共 4 步（描述 → 研究 → 规划 → 实施）

**最佳实践**：
1. **先审查研究**：确保你理解可行性评估
2. **利用发现**：使用研究阶段的技术发现
3. **具体明确**：详细的计划会带来更顺畅的实施
4. **尽早验证**：在实施前审查文档

> **上下文管理**：此规划工作流消耗了大量上下文。为了给实施腾出空间，请运行：
>
> ```
> /compact
> ```
>
> 这将总结对话并保留规划决策，同时减少实施阶段的 token 使用量。
