# Day 1 — 与 Claude Code 的第一次对话

[返回 Day 0（安装设置）](../day0/README.md)

---

你已安装 Claude Code。现在做什么？本指南引导你通过三个级别使用它——每个级别都赋予你对 Claude **如何**工作更多的控制权。

可以把它想象成雇佣一个人：
1. **Prompting** = 向街上的陌生人问路
2. **Agents** = 雇佣一个总是以特定方式做事的专家
3. **Skills** = 该专家针对特定任务有特定的培训

---

## Level 1：Prompting（直接询问）

> 🧠 **想想看**就像给一个知识渊博的朋友发短信。你问"卡拉奇天气怎么样？"，他们会给你*一个*答案——但你不知道他们是查了天气应用、看了窗外、还是凭记忆猜的。

打开终端，输入 `claude`。你现在在对话中。尝试输入：

```
卡拉奇的天气怎么样？
```

Claude 会回答——但**如何**回答是不可预测的。它可能：
- 从训练数据中提取（可能已过时）
- 搜索网络（如果 web 工具可用）
- 给你一个一般性答案而不是实时数据

对于快速提问这完全没问题！但如果你需要**一致、可靠的结果**，仅靠 prompting 是不够的。

### Prompting 什么时候效果很好

- 询问关于代码库的问题（"这个文件是做什么的？"）
- 编写或编辑文档（"把这封邮件重写得更专业"）
- 头脑风暴想法（"为这个活动想 5 个主题行"）
- 解释事物（"用外行的话解释这个错误信息"）

### 局限性

每次你问"天气怎么样？"，Claude 可能以不同的方式获取数据——或者根本不获取实时数据。无法保证它两次使用相同的来源或方法。

## Level 2：Agents（专家）

**Agent** 是扮演特定角色的 Claude——就像分配一个职位头衔。

> 🧠 **想想看**就像一家餐厅的厨房。没有 agent，你走进一个随机厨房大喊"给我做意大利面！"——听到的人可能煮方便面，也可能做五道菜的意大利大餐。有了 agent，你雇佣了一个**意大利面厨师**，他的职位描述写着：*"始终使用新鲜食材，始终煮到 al dente，始终以相同方式摆盘。"*现在你每次都知道得到什么。

这是相同思路应用于 Claude：

> **没有 agent：** 你问 Claude"迪拜天气怎么样？"
> 它可能检查训练数据、搜索网络或猜测。你不知道它会做什么。
>
> **有 agent：** `weather-agent` 有清晰的职位描述：
> *"始终检查迪拜的 Open-Meteo API。始终以特定格式返回温度。"*
> 相同问题，相同方法，每次如此。

### 本仓库的真实示例

本仓库有一个 `weather-agent`——它的全部工作是获取迪拜的温度。以下是它与 just prompting 的区别：

| | Prompting | Agent |
|---|---|---|
| **数据源** | 可能是任何地方 | 始终是 Open-Meteo API |
| **地点** | Claude 随便选 | 始终是迪拜（lat: 25.2, lon: 55.3） |
| **格式** | 随机段落 | 清晰的温度 + 单位 |
| **一致性** | 每次不同 | 相同方法，每次如此 |

### 要点

Agents 给你**可预测性**。相同问题 → 相同方法 → 相同质量。这不是说 agents 更聪明，而是它们**一致**。

## Level 3：Skills（培训）

**Skill** 是 agent（或 Claude 本身）可以使用的特定能力。

> 🧠 **想想看**就像新员工的培训手册。当有人加入你的团队时，他们有一个角色（agent），但他们也要经过特定的培训模块——如何使用 CRM、如何写提案、如何运行 standup。每个培训模块是一个 **skill**。角色告诉他们*是什么*；skills 告诉他们如何做特定的事情。

现在想想一个真人：

> **Shayan** 有很多 skills：
> - 工程 skill——可以写代码
> - 游戏 skill——了解游戏机制
> - 阅读 skill——可以消化和总结长文档
>
> 每个 skill 有自己的知识和方法。Shayan 为合适的任务使用合适的 skill。

Claude 的工作方式相同。`weather-agent` 有一个名为 `weather-fetcher` 的 skill：

- **Agent**（`weather-agent`）= 职位头衔为"Weather Reporter"的人
- **Skill**（`weather-fetcher`）= 关于*如何*获取天气数据的特定培训

Skill 包含精确指令：
1. 调用这个特定的 API URL
2. 从响应中的这个特定字段提取温度
3. 以这个特定的格式返回

### 为什么分离 Agents 和 Skills？

因为**一个 agent 可以有多个 skills**，并且**一个 skill 可以被多个 agents 使用**。

例如，想象你创建：
- 一个 `daily-report-agent` 来总结你的一天
- 它可以同时使用 `weather-fetcher` skill（天气）+ `calendar-reader` skill（会议）+ `email-summarizer` skill（邮件摘要）

Skills 是可重用的构建块。Agents 是使用它们的人。

---

## 综合运用

以下是完整图景：

```
Level 1：PROMPTING
你 → "天气怎么样？" → Claude 以某种方式搞定
                         （不可预测的方法）

Level 2：AGENTS
你 → Weather Agent → 始终使用相同方法
                      （可预测的方法）

Level 3：SKILLS
你 → Weather Agent → 使用 weather-fetcher skill
                      （可预测的方法 + 特定指令）
```

每个级别增加更多控制：

| 级别 | 你控制什么 | 最适合 |
|-------|-----------|--------|
| **Prompting** | 问题 | 快速一次性提问 |
| **Agents** | 问题 + 谁回答 | 可重复的任务 |
| **Skills** | 问题 + 谁回答 + 怎么做 | 关键工作流 |

---

## 接下来做什么？

现在花时间在 **Level 1**——就是 prompting。在终端中习惯向 Claude 提问。你用得越多，就越会发现那些受益于 agent 的任务。
