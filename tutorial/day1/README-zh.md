# 第一天 — 你和 Claude Code 的第一次对话

[返回第 0 天（环境搭建）](../day0/README.md)

---

你已经安装好了 Claude Code。然后呢？本指南带你体验三个层次的使用方式——每一层都让你对 Claude **如何**完成工作有更多控制。

想象一下你在雇人做事：
1. **Prompting（提示）** = 在街上向陌生人问路
2. **Agents（智能体）** = 雇佣一位按固定方式做事的专家
3. **Skills（技能）** = 这位专家针对特定任务接受了专门训练

---

## 第一层：Prompting（直接提问）

> 🧠 **把它想象成** 给一个懂得很多的 friend 发消息。你问 "卡拉奇天气怎么样？" 他们会给你一个答案——但你不知道他们是查了天气 app、看了窗外、还是凭记忆猜的。

打开终端，输入 `claude`。你现在进入了一个对话。试着输入：

```
what is the weather in Karachi?
```

Claude 会回答——但**如何**回答是不可预测的。它可能会：
- 从训练数据中提取（可能已经过时）
- 搜索网络（如果 web 工具可用）
- 给出一个概括性的答案，而非实时数据

对于快速提问来说，这完全没问题！但如果你需要**一致、可靠的结果**，仅靠 prompting 是不够的。

### 什么时候 Prompting 很好用

- 询问有关你代码库的问题（"这个文件是做什么的？"）
- 撰写或编辑文档（"把这封邮件改得更专业一些"）
- 头脑风暴想法（"给我这个活动的 5 个标题建议"）
- 解释说明（"用非开发者也能懂的话解释这条报错信息"）

### 局限性

每次你问 "天气怎么样？" 时，Claude 获取数据的方式可能都不一样——或者根本没有获取真实数据。它无法保证每次使用相同的来源或方法。

---

## 第二层：Agents（专家）

**Agent** 是扮演特定角色的 Claude——就像赋予一个职位名称。

> 🧠 **把它想象成** 一个餐厅厨房。没有 agent 时，你走进一个陌生的厨房大喊 "给我做份意面！"——听到的人可能煮泡面，也可能做一顿五道菜的意大利大餐。有了 agent，你雇佣了一位 **意面主厨**，他的岗位描述写着：*"始终使用新鲜食材，始终煮到弹牙口感，始终以相同方式摆盘。"* 现在你确切知道每次会得到什么。
>
> 同样的思路应用到 Claude 上：
>
> **没有 agent：** 你问 Claude "迪拜天气怎么样？"
> 它可能查训练数据、搜索网络、或者凭经验猜测。你不知道它会怎么做。
>
> **有了 agent：** 一个 `weather-agent` 有明确的岗位描述：
> *"始终查询迪拜的 Open-Meteo API。始终以特定格式返回温度。"*
> 相同的问题，相同的做法，每次如此。

### 本仓库的真实示例

这个仓库有一个 `weather-agent`——它的全部工作就是获取迪拜的温度。以下是它与单纯 prompting 的不同之处：

| | Prompting | Agent |
|---|---|---|
| **数据来源** | 可能是任何地方 | 始终是 Open-Meteo API |
| **地点** | Claude 随便选 | 始终是迪拜（纬度：25.2，经度：55.3） |
| **格式** | 随机的段落 | 清晰的温度 + 单位 |
| **一致性** | 每次不同 | 每次都一样的方法 |

### 要点

Agents 带给你**可预测性**。同样的问题 → 同样的方法 → 同样的质量。这就是优势所在——不是因为 agents 更聪明，而是因为它们**始终如一**。

## 第三层：Skills（专项训练）

**Skill** 是一种特定的能力，agent（或 Claude 本身）可以使用它。

> 🧠 **把它想象成** 新员工的培训手册。当有人加入你的团队时，他们有一个角色（agent），但也要完成特定的培训模块——如何使用 CRM、如何撰写方案、如何主持站会。每个培训模块就是一个 **skill**。角色告诉他们 *是什么*；技能告诉他们 *如何做* 特定的事情。

现在想想一个真实的人：

> **Shayan** 有很多技能：
> - 工程技能 —— 能写代码
> - 游戏技能 —— 了解游戏机制
> - 阅读技能 —— 能够消化和总结长文档
>
> 每项技能都有自己的知识和方法。Shayan 为合适的任务使用合适的技能。

Claude 也是同样的方式。`weather-agent` 有一个名为 `weather-fetcher` 的技能：

- **Agent**（`weather-agent`）= 职位为 "天气播报员" 的人
- **Skill**（`weather-fetcher`）= 关于*如何*获取天气数据的具体训练

这个 skill 包含了精确的指令：
1. 调用这个特定的 API URL
2. 从响应中的这个特定字段提取温度
3. 以这种特定格式返回

### 为什么要把 Agents 和 Skills 分开？

因为**一个 agent 可以拥有多个 skills**，并且**一个 skill 可以被多个 agents 使用**。

例如，假设你创建了：
- 一个 `daily-report-agent`，用于汇总你的一天
- 它可以使用一个 `weather-fetcher` skill（获取天气）+ 一个 `calendar-reader` skill（查看会议）+ 一个 `email-summarizer` skill（总结邮件要点）

Skills 是可复用的构建块。Agents 是使用它们的人。

---

## 融会贯通

以下是完整的图景：

```
第一层：PROMPTING
你 → "天气怎么样？" → Claude 自己想办法搞定
                           （方法不可预测）

第二层：AGENTS
你 → 天气 Agent → 始终使用同样的方法
                     （方法可预测）

第三层：SKILLS
你 → 天气 Agent → 使用 weather-fetcher skill
                     （可预测的方法 + 具体指令）
```

每一层都增加更多控制：

| 层次 | 你控制的内容 | 最适合的场景 |
|-------|-------------|-------------|
| **Prompting** | 问题本身 | 快速的一次性提问 |
| **Agents** | 问题 + 谁来回答 | 可重复执行的任务 |
| **Skills** | 问题 + 谁来回答 + 如何执行 | 关键工作流程 |

---

## 下一步做什么？

现在，先在**第一层**花点时间——就只是 prompt。在终端里熟练地向 Claude 提问。你用得越多，就越能发现哪些任务适合用 agent 来优化。
