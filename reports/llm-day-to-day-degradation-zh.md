# 🔥 Claude Code Ops 4.6 分析。高推理

当 Anthropic 推出 Opus 4.6 这样的模型时，**模型权重**——数十亿个学习参数——已被冻结。训练成本极其高昂（数百万美元，数周的计算时间）。没有人会在夜间重新训练模型。

但权重只是更大系统中的一层。研究揭示，即便模型权重被冻结，至少有 **7 种不同机制** 能够导致真实或感知到的质量变化。

| 问题 | 答案 |
|----------|--------|
| 模型权重在发布后会改变吗？ | **不会**——所有供应商均已确认 |
| 模型是否会日复一日地表现出不同行为？ | **会**——已验证存在 ±8–14% 的方差 |
| 这是故意的"削弱"吗？ | **不是**——没有证据表明存在有意的退化 |
| 基础设施 Bug 真实存在吗？ | **是的**——Anthropic 已确认 3 个 Bug 影响了多达 16% 的请求 |
| 其中是否包含心理因素？ | **是的**——确认偏差和蜜月效应真实存在 |
| 系统提示/训练后调整会改变吗？ | **会的**——各供应商均有记录 |
| 用户应该相信自己的感知吗？ | **部分相信**——真实原因存在，但感知会放大它们 |

## 完整的推理栈

模型权重被冻结，但**其上的九层**可以独立影响你所体验到的结果：

```
┌──────────────────────────────────────────────┐
│  你的会话上下文                              │  ← 在会话内退化
│  (累积的错误、长对话)                         │
├──────────────────────────────────────────────┤
│  系统提示                                    │  ← 定期更新
│  (安全规则、行为指令)                         │
├──────────────────────────────────────────────┤
│  训练后调整 (RLHF / 微调)                     │  ← 可以静默更新
│  (指令遵循、安全对齐)                         │
├──────────────────────────────────────────────┤
│  采样参数                                    │  ← 可在服务端调整
│  (temperature, top-p, top-k)                 │
├──────────────────────────────────────────────┤
│  推测性解码                                  │  ← 草稿模型质量有变化
│  (草稿模型预测 + 验证)                        │
├──────────────────────────────────────────────┤
│  MoE 路由 / 批次组成                         │  ← 已验证存在 ±8–14% 方差
│  (每个请求激活哪些专家)                        │
├──────────────────────────────────────────────┤
│  硬件路由                                    │  ← TPU 对比 GPU 对比 Trainium
│  (哪个集群服务你的请求)                        │
├──────────────────────────────────────────────┤
│  量化级别                                    │  ← 可能在负载下变化
│  (FP16 对比 INT8 对比 INT4 精度)              │
├──────────────────────────────────────────────┤
│  编译器与运行时                              │  ← XLA Bug 已被证实存在
│  (XLA:TPU, CUDA, 硬件特定代码)               │
├──────────────────────────────────────────────┤
│  模型权重 (已冻结)                            │  ← 这些不会改变
│  (数十亿个学习参数)                           │
└──────────────────────────────────────────────┘
```

关键的心智模型：**权重冻结 ≠ 行为冻结**。这就像是说"同样的引擎 = 同样的驾驶体验"，却忽略了轮胎、路况、燃油质量和驾驶员疲劳。

## 已被证实的成因：基础设施 Bug

### Anthropic 2025 年 9 月的事故复盘

2025 年 9 月，Anthropic 发布了一份详细的事故复盘，披露了**三个独立的基础设施 Bug**，这些 Bug 在 2025 年 8 月至 9 月期间降低了 Claude 的质量。他们的官方声明：

> "我们绝不会因为需求、时间段或服务器负载而降低模型质量。用户报告的问题完全是由基础设施 Bug 造成的。"

### Bug #1 — 上下文窗口路由错误

Sonnet 4 的请求被错误地路由到了配置为 1M token 上下文窗口的服务器，而非标准服务器。

- **时间线**：8 月 5 日引入，8 月 29 日在负载均衡变更后恶化
- **峰值影响**：在最严重时段（8 月 31 日），16% 的 Sonnet 4 请求受到影响
- **用户影响**：约 30% 的 Claude Code 用户至少收到了一条降质消息
- **隐蔽细节**：路由具有"粘性"——一旦你命中了故障服务器，后续请求会持续被路由到该服务器
- **修复时间**：9 月 4 日至 18 日（在各平台逐步推出）

### Bug #2 — TPU 输出损坏

TPU 服务器上的配置错误导致 token 生成过程中出现错误，为本应很少出现的 token 分配了高概率。

- **症状**：英文回复中间出现泰文或中文字符，明显的代码语法错误
- **影响范围**：Opus 4.1 和 Opus 4（8 月 25–28 日），Sonnet 4（8 月 25 日至 9 月 2 日）
- **范围**：仅限 Claude API；第三方平台不受影响
- **修复时间**：9 月 2 日回滚

### Bug #3 — XLA:TPU 编译器误编译（最棘手的）

一项修复精度问题的代码变更意外地触发了 Google XLA:TPU 中一个**潜伏的编译器 Bug**。

- **根本原因**：近似 top-k 操作（用于选择最可能的下一个 token）"有时会返回完全错误的结果，但仅针对某些批次大小和模型配置"
- **难以发现的原因**：其行为会随前后执行的操作以及调试工具是否启用而变化
- **隐藏数月**：2024 年 12 月的一个先前临时修复曾意外地掩盖了这个更深层的 Bug
- **影响范围**：确认影响 Haiku 3.5；怀疑影响部分 Sonnet 4 和 Opus 3
- **解决方案**：从近似 top-k 切换为精确 top-k；接受了"轻微的效率影响"，因为"模型质量不可妥协"

### 检测为何困难

Anthropic 自身的自动化评估未能捕捉到用户报告的退化，"部分原因是 Claude 通常能从孤立错误中良好恢复。"每个 Bug 在不同平台以不同速率产生不同症状，形成了"一堆令人困惑的混杂报告，并未指向任何单一原因。"

关键背景：Claude 运行在**三种不同的硬件平台**上（AWS Trainium、NVIDIA GPU、Google TPU），每种平台具有不同的故障模式、编译器和精度行为。你的请求在不同日期可能会命中不同的硬件。

## 已被证实的成因：MoE 路由方差

现代大型模型通常采用**混合专家（MoE）**架构，其中仅有一部分模型参数（"专家"）针对每个输入被激活。一个经过学习的路由器决定使用哪些专家。

Scale AI 的研究揭示了一个关键发现：

> "稀疏 MoE 与批处理推理的结合会产生不可预测的结果，因为批次的组成决定了你的查询被路由到哪个专家，而同一批次中其他用户的查询组合是非确定性的。"

### 各供应商的日常方差测量值

| 供应商 | 日常分数方差 |
|----------|--------------------------|
| OpenAI（GPT-4 变体） | ±10–12% |
| Anthropic（Claude 变体） | ±8–11% |
| Google（Gemini 变体） | ±9–14% |

具体例子：同一个模型在某天对越狱防御的得分是 **77%**，而第二天是 **63%**。同一个模型，相同的权重，相同的测试——仅因基础设施导致的波动就达到 14 个百分点。

这意味着即使零 Bug、零变更，同一个模型在不同日期仅因请求的批处理和路由方式不同，就可能产生明显不同的质量输出。当日常噪声为 10–15% 时，A/B 测试无法可靠地检测出 5% 的质量信号。

## 已被证实的成因：系统提示与训练后调整更新

### 系统提示变更

模型权重不变，但包裹这些权重的**系统提示**可以随时更新。对 Claude 系统提示演变过程的分析显示有数十次迭代，其中"热修复"——为修补不良行为而添加的简短指令——被定期添加和移除。

Claude 3.7 的系统提示包含多条针对常见 LLM"陷阱"的热修复指令。Claude 4.0 的系统提示移除了所有这些指令，相关问题行为通过强化学习在训练后阶段得到处理。

### 训练后调整理论

对无法解释的质量转变最合理的理论：公司可以在不改变基础模型权重的情况下更新**微调和 RLHF**（基于人类反馈的强化学习）。这在技术上使得"模型没有改变"的说法成立，同时仍能通过更新安全护栏和指令遵循调整来改变行为。

## 已被证实的成因：静默模型替换

OpenAI 已被多次记录在案，静默地更换用户所交互的模型：

- 一夜之间移除模型选择器，强制用户从 GPT-4o 切换到 GPT-5
- 将 GPT-4o 变成一个隐藏的"旧版模型"，需要在设置中手动切换，且应用内无任何通知
- 一个"自动切换器" Bug 将用户路由到错误的模型
- Plus 订阅用户报告模型在未经同意的情况下切换到"受限版本"

Sam Altman 承认这次发布"比我们预期的要坎坷一些。"Reddit 帖子获得了数千个赞，称新模型是一场"灾难"和"降级"。

这表明模型替换**确实会**在行业中出现——有时是有意的（产品决策），有时是意外的（路由 Bug）。

## 促成因素

### 负载下的量化

为了经济高效地服务数百万用户，公司可能会提供模型的**量化**版本——将精度从 FP16 降低到 INT8 或 INT4。这可以将内存使用减少 2–4 倍并加速推理，但会引入微小的质量损失。供应商是否会在负载下动态切换量化级别尚有争议，但该技术能力是存在的，并且在 vLLM 和 TensorRT 等服务框架中有详细记录。

### 推测性解码

现代服务栈使用一个较小的"草稿"模型来预测多个后续 token，然后让真实模型验证它们。理论上这保持了相同的输出分布，但在实践中，接受率因领域和上下文而异。开箱即用的草稿模型在某些情况下可能表现良好，但在领域特定任务或极长上下文中往往会遇到困难。

### 上下文窗口污染

在长编码会话中，早期错误会在上下文中累积。模型会看到自己的错误并可能延续它们。这是单个会话中"Claude 变笨了"最常见的原因——并非模型在退化，而是上下文受到了污染。

**实用建议**：当感觉质量不佳时，使用 `/compact` 或开启全新的会话。这是你能做的最具可操作性的一件事。

## 斯坦福研究——以及为何它很复杂

2023 年斯坦福大学和加州大学伯克利分校的标志性研究（Chen, Zaharia, Zou）——"ChatGPT 的行为如何随时间变化？"——经常被引用来证明 LLM 会退化。其头条发现：

> GPT-4 在"这个数字是质数吗？逐步思考"问题上的准确率从 **97.6%** 下降到 **2.4%**（2023 年 3 月至 6 月间）。

### 该研究证明了什么

- 同一 LLM 服务的**行为可以在短时间内发生显著变化**
- 不同能力可能朝相反方向变化（GPT-4 数学变差，GPT-3.5 变好）
- 代码生成质量下降（GPT-4 可执行代码：52% → 10%）
- 该研究提出了术语 **"LLM 漂移"**

### 方法论批评

- 三月的版本使用了 **temperature 0.0**，而六月的版本使用了 **temperature 1.0**——这是一个根本性的混淆变量，会增加随机性
- 每个任务仅 **500 次查询**——对于确定性的统计主张来说样本量太小
- 所谓的"数学问题"实际上是是/否问题，模型只是改变了猜测模式，而非数学能力发生了变化
- 这些变化可能反映了有意的**训练后安全更新**，而非退化

该研究证明了一些重要的事情——LLM 行为随时间变化——但其机制很可能是有意的更新，而非无意的退化。

## 心理因素

### 确认偏差

一旦有人发推说"今天的 Claude 很笨"，你就会开始注意到每一个错误。在没有人抱怨的日子里，你会对同样的错误不以为意。社交媒体放大了这种效应。

### 蜜月效应

用户对新模型会经历一个初始的蜜月期，然后逐渐发现限制。模型没有改变——期望值的提升速度超过了能力本身的提升速度。

### 任务难度变化

你的任务每天都在变化。一天处理困难问题会感觉模型变差了，即使它并没有。

### "周末 Claude" 迷思

尽管许多用户相信存在星期几相关的质量模式，但严谨的分析发现**没有一致的证据**支持星期几相关的质量模式。一篇题为"AI 在周一更笨"的分析最终一无所获。

### LLM 的随机性本质

LLM 是概率性的。相同的提示每次可能产生不同的输出。在运气不佳时，你可能会连续收到几个较差的回复——这纯粹是随机性，而非退化。

---

## 结论

用户所描述的现象是**真实的，但归因有误**：

- **正确**：他们在某些日子的体验确实有所下降
- **错误**：模型被有意地"削弱"了

实际原因是由以下因素共同导致：

1. **基础设施 Bug**——Anthropic 的事故复盘已证实（高达 16% 的请求受影响）
2. **MoE 路由方差**——Scale AI 测量到 ±8–14% 的质量波动，即使零变更
3. **系统提示与训练后调整更新**——各供应商均有记录
4. **硬件异构性**——TPU 对比 GPU 对比 Trainium，每种具有不同的故障模式
5. **上下文污染**——长会话会降低会话内质量
6. **确认偏差**——社交媒体放大了感知到的模式
7. **随机性方差**——相同模型、相同提示、每次输出不同

测量问题非常严重：日常方差为 ±8–14%，意味着你无法区分真实的 5% 质量变化与噪声。这就是为什么"这一切都是你的幻觉"和"他们削弱了模型"两派都对自己的观点充满信心的原因——信噪比使得仅凭个人体验无法判断。

# 🔥 Codex 5.3 高推理与发现

### 报告范围

本节解释为何用户可能在短时间内体验到 Claude 输出质量下降，而 Codex 5.3 在编码任务上感觉稳定或更强。重点不是永久的模型质量排名。重点是真实服务条件下的短期生产行为。

报告日期：2026 年 3 月 5 日。

### 观察到的模式

报告的典型模式是：

1. 模型质量在一段时间内可接受。
2. 质量在数天内似乎下降。
3. 质量恢复到接近此前的基线水平。

这种形态通常是服务栈或发布模式，而非永久性的基础模型能力变化。永久性的能力下降通常不会在没有显式回滚或修复的情况下如此迅速地恢复。

### 高推理：为何 Codex 5.3 在糟糕窗口期可能看起来更好

在另一供应商的降质期间，Codex 5.3 可能因几个技术原因（这些原因可能同时发生）而显得明显更强：

1. **产品目标契合度**。Codex 5.3 针对代码生成和 agentic 编码工作流进行了优化，因此即使原始模型能力相当，由于工具编排、仓库推理和以代码为中心的指令调优，也能产生更好的编码结果。
2. **推理策略差异**。供应商独立调整延迟、推理深度和解码默认值。同一天，一家供应商更保守的策略可能看起来比另一家激进的、速度优化的策略"更聪明"。
3. **服务路径分离**。即使两家供应商都托管了最先进的模型，它们运行不同的路由层、编译器/运行时栈和发布管道。一个栈中的事故并不意味着另一个栈存在相关的退化。
4. **发布与回滚时机**。如果一个供应商处于发布中期而另一个供应商保持稳定，用户可能会观察到巨大的临时质量差异，而基础模型权重并无长期变化。
5. **会话级污染效应**。在长编码对话中，错误累积会放大感知到的下降。竞争对手的助手可能仅仅因为故障会话被重置，或其工具循环恢复得更快，而感觉更好。

### 详细发现

对于像"Claude 在大约四天内感觉很弱，然后又恢复了"这样的报告，最可能的解释是：

1. 供应商侧的事故、路由问题、解码/运行时 Bug 或发布回归影响了一部分请求。
2. 问题持续了足够长的时间，以致在实际工作流中被反复注意到。
3. 问题被修复或回滚。
4. 感知到的质量迅速恢复。

在同一时期，Codex 5.3 可能感觉好得多，因为它不共享相同的事故路径，并且编码任务的优化放大了实际结果上的差距。

### 针对该模式的假设排名

| 假设 | 可能性 | 理由 |
|------------|------------|-----------|
| 供应商事故加回滚 | 高 | 最符合持续数天的下降后快速恢复的模式 |
| 服务配置变更（采样/延迟/推理预算） | 高 | 行为突然变化的常见来源，无需模型重新训练 |
| 静默别名或快照变动 | 中高 | 可在用户无可见操作的情况下改变行为 |
| 仅提示漂移和上下文污染 | 中 | 可能降低会话质量，但不太可能单独解释广泛的、持续数天的报告 |
| 永久性基础模型退化 | 低 | 与快速恢复到之前质量水平不一致 |

### 什么会证实或证伪这一发现

要将这一发现从高置信度推断转化为确凿证据，需要收集跨多天相同任务集的请求级遥测数据：

1. 请求时的精确模型标识符和快照/别名。
2. 供应商暴露的任何后端指纹或发布标记。
3. 解码参数（temperature, top_p, top_k, max tokens）。
4. 延迟、超时和错误率追踪。
5. 在固定编码基准提示集上的结构化质量评分。
6. 失败点的会话长度和 token 上下文深度。

如果质量下降与事故窗口、配置变更或后端指纹变化相关，则事故/配置假设被证实。如果不存在此类变化且退化仅发生在长会话中，则上下文污染成为主要解释。

### 实用工程指导

为减少生产中的日常方差：

1. 在可用时固定模型快照，而非使用浮动别名。
2. 存储请求元数据（模型 ID、参数、延迟、错误、响应质量标签）。
3. 针对编码任务运行每日固定金丝雀测试套件，并在回归时发出警报。
4. 在连续多次失败后重置或压缩长时间运行的会话。
5. 为事故窗口保留备用供应商/模型路径。
6. 在内部仪表板中区分"模型质量"和"服务可靠性"。

### 最终结论

Codex 5.3 在 Claude 短期降质窗口期表现更好，是现代 LLM 运营中技术上合理且符合预期的结果。最强的解释不是永久性的模型崩溃。最强的解释是一个供应商的临时服务路径退化，加上同一时期另一供应商的编码特定优化和稳定运营。
