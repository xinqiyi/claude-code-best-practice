# 为什么 Harness 至关重要

为什么 Claude Code 的功能并非"只是乔装打扮的提示词"——以及为什么 harness 才是真正将玩具级输出与生产级工程工作区分开来的关键。

## 执行摘要

Claude Code 资深用户中一种常见的化约论观点是：*"skills、commands、subagents、hooks——它们最终都会变成给模型的提示词，所以一个强大的提示词本身就足以替代它们。"*

在模型最终推理调用的层面，这在技术上是正确的。模型只看到 token。

在所有其他层面——也就是真实软件工程发生的层面——**这种化约论站不住脚。**Harness 不是一个提示词投递系统。它是一个**提示词构建系统、一个确定性执行系统、一个上下文架构系统**——而这些能力是无法用更强大的措辞来替代的。

本报告阐述了这种化约论哪里正确、哪里失效，以及为什么混淆"模型看到了什么"与"系统做了什么"会让实践者远离那些真正赋予 Claude Code 杠杆效应的功能。

---

## 听起来有道理的化约论

对于一个**单次原子任务**——"给我写一个递归斐波那契函数"——harness 对输出质量没有任何贡献。将相同的 token 交给同一个模型，无论它们是通过 skill、command 还是原始提示词传入的，你都会得到相同的输出分布。

在这个狭窄的范围内，化约论成立：

> 输出质量 ≈ 提示词质量

这是 Claude Code 相比普通聊天机器人几乎没有额外价值的范围。这也是化约论隐含假设的范围——恰恰也是真实工程工作并不处于的范围。

---

## 化约论在哪些方面失效

Harness 的十项架构能力运行在提示词无法触及的层面。

| # | 能力 | 作用 | 提示词为何无法复制 |
|---|------------|--------------|-------------------------------|
| 1 | **上下文隔离** | Subagents 在独立的上下文窗口中运行 | 一个提示词填满一个窗口。N 个并行的 subagent 提供约 N× 的有效上下文。 |
| 2 | **Harness 强制执行的工具限制** | `allowed-tools` / `disallowedTools` 在模型使用工具之前就将其屏蔽 | 提示词中的指令是建议性的；模型可能忽略它们。拒绝规则无法被忽略。 |
| 3 | **懒加载规则与记忆** | `paths:` frontmatter 和子目录下的 `CLAUDE.md` 文件仅在 Claude 触及匹配路径时加载 | 提示词是静态的——它无法根据运行时读取了哪些文件来有条件地加载内容。 |
| 4 | **Hooks：确定性代码执行** | Shell 命令在生命周期事件（PreToolUse、PostToolUse、Stop 等）时执行，并且可以**阻止**工具调用 | 提示词无法拦截自身的工具调用。即使模型不"希望"hooks 执行，它们仍然会执行。 |
| 5 | **模型路由** | `model: haiku` 或 `model: opus` 将调用路由到不同的模型端点 | 提示词中的任何 token 都无法改变哪个模型来回答。 |
| 6 | **并行性** | 多个 subagent 同时执行 | 提示词是串行的。Harness 调度并收集来自多个并行进程的结果。 |
| 7 | **跨会话持久化** | 记忆系统和设置层级结构在多次对话间持久保存 | 提示词在会话结束时消亡。 |
| 8 | **模块化系统提示词** | CLI 根据已激活的功能，有条件地加载 110 多个系统提示词片段 | 用户无法手动编写或替换 CLI 的内部提示词片段。 |
| 9 | **Skill 预加载** | `skills:` 字段将一个 skill 的全部内容注入到 subagent 的初始上下文中 | 用户无法预先填充另一个 agent 的上下文——只有 harness 加载器可以做到。 |
| 10 | **权限分类** | `auto` 权限模式使用后台分类器来预先批准或阻止工具调用 | 提示词无法为自身添加执行前的安全层。 |

每一行都是一个维度，在这些维度上，"更强的措辞"根本无法替代。

## "提示词"这个词的两种用法

化约论利用了词语歧义。"提示词"一词被用来指代两种截然不同的事物：

| 含义 | 谁控制它 | 大小 |
|---------|-----------------|------|
| (a) 用户输入的内容 | 用户 | ~6–60 tokens |
| (b) 模型在推理时看到的内容 | Harness | ~5,000–50,000+ tokens |

在聊天机器人中，(a) 和 (b) 是同一回事。
在 Claude Code 中，它们截然不同。

Harness 的工作正是让 (b) 远丰富于 (a)：

```
用户输入："实现一个递归 flatten 函数"         ← (a) ~6 tokens

模型在推理时实际看到的内容：                   ← (b) ~15,000 tokens
  ├── CLAUDE.md（项目约定）
  ├── 匹配的 .claude/rules/*.md（通过 paths: frontmatter 加载）
  ├── 模块化系统提示词片段
  ├── 工具定义
  ├── 环境上下文（cwd、git status、平台）
  ├── 之前的对话历史
  ├── 模型通过 Read/Grep 工具读取的文件
  └── 用户的 6 token 请求
```

**输出质量是 (b) 的函数，而不是 (a) 的函数。** Harness 构建了 (b)。"仅靠强大的提示词"无法复现 (b)，因为它的大部分内容并非由用户编写。

## 即使仅论输出质量，Harness 也在发挥作用

考虑相同的提示词——"实现一个递归 flatten 函数"——在三种环境中：

| 环境 | 模型看到的内容 | 典型结果 |
|-------------|---------------------|----------------|
| 聊天机器人，无工具 | 一句话 | 教科书式递归，通用风格 |
| Claude Code，未读取文件 | 句子 + CLAUDE.md | 匹配声明的项目约定 |
| Claude Code，自主循环 | 句子 + CLAUDE.md + 读取相邻文件 + 运行测试 | 匹配实际代码库模式，通过测试，处理现有代码已处理的边界情况 |

相同的模型。相同的用户提示词。**三种不同的输出质量。** 差别在于 harness——具体来说，是它组装的**有效上下文**以及它启用的**迭代循环**。

对于非 trivial 的任务，输出质量是以下因素的函数：

```
输出质量 = f(有效上下文, 模型能力, 迭代循环)
```

用户控制着*有效上下文*的一小部分（他们输入的提示词）。Harness 控制着其余部分——以及整个迭代循环。

## 化约论哪里对了（以及哪里错了）

| 主张 | 判定 |
|-------|---------|
| "在推理时，模型只看到 token。" | ✅ 正确 |
| "Skills、commands 和 subagent 提示词都为某些上下文贡献了 token。" | ✅ 正确 |
| "对于真空中的原子任务，提示词质量主导输出质量。" | ✅ 正确 |
| "因此，一个强大的提示词等同于使用各种功能。" | ❌ 错误 |
| "因此 harness 对输出质量无关紧要。" | ❌ 在真实工程任务中错误 |

前三句话是准确的观察。然而跳跃到第四句时，推理就失效了：它将模型与包裹它的系统混为一谈，并将原子任务与真实工程工作混为一谈。

---

## 正确的心智模型

> **提示词控制的是模型被要求做什么。**
> **Harness 控制的是系统在模型无法触及的层面做什么**——在 token 到达之前、在 token 产生之后、跨会话、跨上下文、跨进程。

功能不是加了步骤的提示词。它们是 **harness 层面的原语**——确定性执行、上下文架构和基础设施路由——运行在模型没有发言权的层面。

一个有用的类比：

| 层面 | 聊天机器人 | Claude Code |
|-------|---------|-------------|
| 菜谱 | 用户的消息 | 用户的消息 + harness 组装的上下文 |
| 厨房 | 无——只是一个学生 | 工具、hooks、记忆、并行工作者、生命周期事件 |

你可以写出全世界最好的菜谱。但没有厨房，你就无法大规模烹饪。

---

## 给实践者的要点

1. **对于原子问题，提示词质量几乎就是一切。** Harness 无关紧要。如果你只需要这个，直接用聊天机器人。
2. **对于真实的代码库工作，harness 在默默地承担繁重的工作。** 推理时的有效提示词大部分是由 harness 构建的，而非用户输入的。
3. **善用那些提示词绝对无法做到的功能：** 确定性（hooks）、隔离（subagents）、懒加载（带有 `paths:` 的 rules）、持久化（memory）、路由（每个 agent 的 `model:`）和并行（parallelism）。
4. **强大的提示词是必要条件，但不是充分条件。** 功能为你提供了提示词无法实现的确定性、隔离性和组合性。两者是互补关系，而非替代关系。
